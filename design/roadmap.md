## 0. 이 문서를 읽는 법 (문외한용)

게임은 **한 번에 다 만들지 않는다.** 작게 만들어 "재미있나" 확인하고 → 한 구간을 예쁘게 다듬어 "팔리겠나" 확인하고 → 전체를 완성해 출시하고 → 라이브로 키운다. 이 순서가 **로드맵의 척추**이고, scope-mvp.md가 정한 4단계가 그대로 4개의 큰 정거장이다.

```
   [지금]
     │
  ┌──▼─────────┐   ┌──────────────┐   ┌──────────┐   ┌───────────┐
  │ 1. 프로토타입 │──▶│ 2. 수직 슬라이스 │──▶│ 3. v1.0   │──▶│ 4. 출시 후 │
  │ "재미있나?"  │   │ "팔리겠나?"     │   │ "완성"    │   │ "키운다"   │
  └────────────┘   └──────────────┘   └──────────┘   └───────────┘
   재미 검증         스토어 GIF          Steam 출시      라이브 콘텐츠
```

각 정거장에는 **졸업 기준**이 있다. 그걸 통과해야 다음으로 간다. 통과 못 하면 — 다음으로 가지 말고 **그 자리에서 고친다.** 게이트의 존재 이유가 그것이다(특히 정거장 1).

**핵심 원칙 하나:** 정거장 1에서 만드는 **기술 코어(저장·큰 수·게임 루프)는 모든 정거장이 재사용한다.** 그래서 1일차부터 제대로 짜야 하고, 그게 이 로드맵에서 가장 중요한 "한 번만 제대로" 구간이다.

## 1. 정거장 1 — 프로토타입 ("코어 루프가 재미있나?")

> **목표:** 못생긴 채로, 압축→작아짐→강해짐의 루프가 손에서 돌고 첫 상전이가 "또 하고 싶다"를 만드는지 확인. **남에게 보여주는 게 아니라 우리가 확인하는 단계.**

### 1-A. ★ 1일차부터 고정할 것 (tech-architecture 기반 — 나중 도입 비용이 큼)

이 4가지는 **프로토타입 첫날 코드 골격에 박아넣는다.** 나중에 추가하면 이미 쌓인 세이브·코드를 깨야 한다(tech-architecture §1.2·§7 후속 메모 명시). 프로토타입이 "임시"라도 이 뼈대만은 영구적이다.

| # | 1일차 고정 | 왜 지금 (안 하면) | 출처 |
|---|---|---|---|
| F1 | **세이브 봉투 포맷 `{version, data, checksum}`** | 나중에 `version`을 추가하면 그 전 세이브들은 마이그레이션 불가 → 영영 못 읽음 | tech-arch §1.2 |
| F2 | **StorageAdapter 추상화 (read/write/exists/backup)** | 웹 localStorage로 시작해도 인터페이스로 감싸야 Steam(fs) 이식 시 게임 코드 0줄 수정 | tech-arch §1.6 |
| F3 | **고정 timestep 게임 루프 (tick/render 분리)** | economy 시뮬이 dt=1~2s 결정적 가정. 나중에 바꾸면 검증된 페이싱이 실제 게임에서 안 맞음 | tech-arch §4.1 |
| F4 | **break_eternity.js 경계 (게임수치=Decimal, 카운트=number)** | break_infinity로 시작하면 재하강 누적 lifetime_C가 한계 돌파 → 교체 마이그레이션 비용. 처음부터 break_eternity | tech-arch §2.1·§2.2 |

> 이 4개를 "F-게이트"라 부른다. **F-게이트 없이 콘텐츠부터 만들기 시작하면 안 된다.** 이건 집의 기초공사다.

### 1-B. 마일스톤 (대략 순서)

| 순서 | 마일스톤 | 핵심 산출물 | 규모 |
|---|---|---|---|
| M1.1 | **F-게이트 + 빈 게임 루프** | 봉투 세이브·StorageAdapter·고정 timestep·Decimal 경계가 도는 빈 껍데기. 화면엔 숫자 하나 | 중간 (한 번만 제대로) |
| M1.2 | **8단 압축기 체인 + 코어 수식** | dec0~19 진행, base_k·growth·dec=α·log₁₀C, r↓+E↑. 자기증식 가속 | 중간 |
| M1.3 | **알려진 물리 온보딩 (dec0~9)** | 분자→원자→핵→핵자→쿼크 무상전이 층 진입 + "층 진입 ≠ 상전이" | 작음 (수치 대부분 확정) |
| M1.4 | **능동 손맛: 수동 압축 + 오비탈 공명** | 클릭 피드백, 공명 슬롯, 60/40 룰 | 작음~중간 |
| M1.5 | **첫 상전이 (PT1, dec19, 프리온 진입)** | QF 획득 → production_mult 부스트. "프레스티지의 무게" | 중간 |
| M1.6 | **미지 서브층 1~2개 (프리온 + 선택 끈)** | 위상 겹침 메커니즘 (+진동 하모닉스) | 중간 |
| M1.7 | **도감 일부 + 연구 1~2가지 + 오프라인** | 알려진 물리 ~15~25입자, 체인증폭 A가지 일부, 오프라인 크레딧(24h캡·0.65·탬퍼클램프) | 중간 |

- **순서 근거:** F-게이트(M1.1)가 모든 것의 토대 → 코어 엔진(M1.2)이 일의 대부분 → 온보딩·손맛(M1.3~4)이 첫 경험 → 첫 상전이(M1.5)가 클라이맥스 → 미지/도감/오프라인(M1.6~7)이 "방치형+미지"의 증거. 데이터 주도(tech-arch §4.4 `data/`)라 입자·노드 수치는 economy/content 확정분을 주입만.
- **병렬 가능:** M1.4(손맛)와 M1.7의 도감 데이터는 코어 엔진과 독립적이라 1인이라도 "코어 막힐 때 도감 작업" 식으로 전환 가능. 팀이면 분담.

### 1-C. 졸업 기준 (이게 충족돼야 정거장 2로)

scope-mvp §2 졸업 기준 그대로:
1. 첫 30초에 "작아짐=강해짐"이 화면에서 느껴진다.
2. 자동화 전 1~2분 능동 클릭이 지루하지 않다.
3. 첫 상전이가 "해냈다 + 또 하고 싶다"를 만든다.
4. 닫고 켜니 오프라인 진행이 기분 좋다.
5. **개발자 본인이 "한 판 더" 하고 싶다.** (가장 중요)

> ⚠️ **3·5가 안 나오면 콘텐츠를 더 만들 때가 아니라 코어 루프를 고칠 때다.** 여기서 멈추고 economy/systems를 다시 본다. qa-balancer 플레이테스트로 객관 확인 권장.

### 1-D. 의존성
- **선행:** 없음 (출발점). economy/systems/codex 확정 수치는 이미 존재 → 주입만.
- **이 단계가 막는 것:** 정거장 2·3 전부. 코어가 안 돌면 다음이 없다.

## 2. 정거장 2 — 수직 슬라이스 ("진짜 게임처럼 보이고 들리나?")

> **목표:** 한 구간(온보딩~첫 상전이 추천)을 비주얼·사운드·게임필까지 *완성도*로 다듬어 **스토어 GIF·트레일러 클립이 나오는 수준**으로. 폭이 아니라 깊이.

### 2-A. 마일스톤

| 순서 | 마일스톤 | 핵심 산출물 | 규모 |
|---|---|---|---|
| M2.1 | **아트 디렉션 1차 확정** | 층 팔레트·무드, 파티클 미감, r 시각화 스타일 (art-director) | 중간 |
| M2.2 | **사운드 1차** | 압축 클릭·공명·상전이 사운드 (Criticality식 청각 피드백 — 반복 클릭 만족감) | 중간 |
| M2.3 | **스케일 전환 VFX** | 층 진입 줌 트랜지션 + 첫 상전이 연출 (graphics-programmer) | 중간 |
| M2.4 | **FTUE 풀 폴리시 (첫 5분)** | 텍스트 벽 없이 점진 공개, 온보딩~첫 상전이 (ux-designer) | 중간 |
| M2.5 | **내러티브 보이스 적용** | 그 구간 층 전환 비트 + 첫 상전이 메시지 (narrative 확정분) | 작음 |
| M2.6 | **게임필/주스** | 숫자 카운트업, 글로우, 마이크로 애니메이션 | 작음~중간 |

- **순서 근거:** 아트 방향(M2.1)이 사운드·VFX의 톤을 결정 → 동시 진행 가능한 사운드(M2.2)·VFX(M2.3) → FTUE 폴리시(M2.4)가 이 모두를 첫 경험에 엮음 → 보이스·주스(M2.5~6)가 마감.
- **병렬:** art-director / audio-designer / graphics-programmer가 분리 레인 → **정거장 1 코어를 안 건드리므로 대부분 병렬.** (audio-designer가 현재 사운드 문서 병렬 작성 중 = 이 정거장의 입력 준비.)

### 2-B. 졸업 기준
1. 그 구간만 떼어 보여줘도 "완성된 게임" 같다.
2. **스토어용 GIF 3~5개가 이 슬라이스에서 나온다** (→ 정거장 3에서 위시리스트 시작점, scope §3 연결).
3. 압축 클릭·상전이 사운드가 "기분 좋다".
4. 첫 5분이 설명서 없이 이해된다.

### 2-C. 의존성
- **선행:** 정거장 1 졸업(코어 루프가 돈다). 폴리시할 *내용*이 있어야 폴리시한다.
- **산출의 쓰임:** 이 슬라이스의 GIF가 정거장 3의 **launch-plan 위시리스트 캠페인 출발점**(launch-plan.md §위시리스트).

## 3. 정거장 3 — v1.0 Steam 정식 출시 ("완성된 게임")

> **목표:** 온보딩부터 첫 빅 크런치를 지나 재하강 루프 안착까지, 4필러가 끝까지 작동하는 완전한 1차 사이클 + 반복의 입구. scope-mvp §4 IN 목록 전부.

### 3-A. 마일스톤 (대략 순서)

| 순서 | 마일스톤 | 핵심 산출물 | 규모 |
|---|---|---|---|
| M3.1 | **미지 서브층 4개 추가** | 루프(스핀네트워크)/거품(요동)/정보(홀로그래픽)/플랑크(픽셀붕괴). 6 미지 메커니즘 완성 | 큼 |
| M3.2 | **빅 크런치 (PT7) + "시작점" 프레이밍** | 플랑크 도달→최종 QF 폭발→재하강 입구. 종착점 언어 금지 | 중간 |
| M3.3 | **재하강 기본 루프 (1~2회)** | dec0 재시작 + lifetime_C·QF 누적, 회전 집중 서브층 선택, 방치/개입 분리, D보존 회차곡선 + 심화 훅 씨앗 1~2개 | 큼 |
| M3.4 | **도감 87입자 완성 + 홀로그래픽 배율** | 실제 57 + 상상 30, 곡선 B(≤×1.35) | 큼 |
| M3.5 | **연구 트리 핵심 가지** | A.체인증폭(14)+C.자동화(10) 전체 + B/D 코어 노드 | 중간~큼 |
| M3.6 | **아트·사운드·내러티브 전체 확장** | 6층 팔레트, 미지 메커니즘 VFX, 재하강 연출(회차별 축소), 풀 보이스 | 큼 |
| M3.7 | **Steam 패키징 + 인프라** | NW.js+steamworks.js, Steam Cloud, 오버레이, 업적, lz-string 세이브 | 중간 |
| M3.8 | **FTUE·설정·접근성·세이브 안전 + 텔레메트리 수집** | 옵션, 시계조작 방어, 롤링 백업, D_total 텔레메트리 *수집* 코드(scope C4) | 중간 |
| M3.9 | **출시 폴리시 + QA + 밸런스 패스** | 버그·진행막힘·세이브 손상 점검, 페이싱 재검증 (qa-balancer) | 중간~큼 |

- **순서 근거:** 콘텐츠 확장(M3.1~5)이 본체 → 폴리시(M3.6)가 그 위에 → 패키징/인프라(M3.7~8)는 콘텐츠와 병렬 가능(platform 레이어 격리, tech-arch §5.2) → 최종 QA(M3.9)가 마감. **재하강(M3.3)은 빅 크런치(M3.2) 의존.**
- **컷 안전판:** 시간이 부족하면 M3.5 연구 D가지 후반 노드부터 출시 후로(scope C3). M3.3 재하강은 1회만 확실히 돌아도 출시 가능(2회차는 라이브 보강 가능하나 권장은 1~2회).
- **텔레메트리 주의 (scope C4 / risk R8):** holo_factor 재튜닝은 출시 후지만, **수집 코드(M3.8)는 v1.0 IN.** 안 넣으면 출시 후에도 데이터가 없어 영영 보정 불가.

### 3-B. 졸업 기준 (= 출시 가능)
1. 첫날 첫 상전이 → ~1주 첫 빅 크런치 → 재하강 1~2회로 "수 주" 진입까지 4필러가 한 번도 끊기지 않는다.
2. 6 미지 메커니즘이 각각 "새롭다" (필러 ④ 끝까지).
3. 도감 87 완성 가능, 홀로그래픽 배율이 가드레일(≤×1.35) 안.
4. 세이브가 안전하다 (손상 복구·마이그레이션·백업 작동 — tech-arch §1).
5. Steam 빌드가 오버레이·클라우드·업적과 함께 돈다.
6. qa-balancer 출시 전 체크리스트 통과 (launch-plan §출시 전 체크리스트).

### 3-C. 의존성
- **선행:** 정거장 2 졸업(폴리시 파이프라인 확립). 정거장 2의 아트/사운드 방향을 6층 전체로 확장하는 것이 M3.6.
- **병렬:** 위시리스트 캠페인(launch-plan)은 정거장 2 GIF 확보 시점부터 *이미 진행 중*이어야 함 → 정거장 3 내내 마케팅과 제작이 병행.

## 4. 정거장 4 — 출시 후 (Post-launch) ("라이브로 키운다")

> **목표:** 출시한 게임을 *넓혀* 리텐션·재방문 동기를 만든다. scope-mvp §5 OUT 목록을 라이브 콘텐츠로.

### 4-A. 우선순위 (출시 후 데이터 보고 조정)

| 순위 | 항목 | 규모 | 근거 |
|---|---|---|---|
| 1 | **재하강 심화 콘텐츠 (~18입자+~12노드)** | 큼 | scope C1. 회차마다 다른 깊이 보상. 가장 큰 라이브 콘텐츠 |
| 1 | **D_total 텔레메트리 기반 holo_factor 재튜닝** | 작음 | scope C4 / risk R8. 실데이터 도착 후 1.x 패치 밸런스 보정 |
| 2 | **3회차+ 재하강 차별화 / 후기 회차 변별** | 중간 | scope C2. 텔레메트리로 권태 지점 확인 후 튜닝 |
| 2 | **연구 트리 심화 가지 (D후반·B일부)** | 중간 | scope C3. 컬렉터·하드코어용 |

- **선행:** v1.0 출시 + 텔레메트리 데이터 수집(M3.8). 라이브 튜닝은 실플레이어 데이터 없이 불가.

## 6. 소규모·1인 팀을 위한 현실 조언 (문외한용)

- **순서를 지켜라.** 정거장 1을 건너뛰고 콘텐츠(정거장 3)부터 만들고 싶은 유혹이 가장 크고 가장 위험하다. 재미를 모르는 채 87입자를 만들면 다 버릴 수 있다.
- **F-게이트(1일차 고정)는 타협하지 마라.** "나중에 제대로" = 세이브를 깨고 다시 만든다는 뜻. 이건 인디가 흔히 겪는 가장 비싼 후회다.
- **막히면 레인을 바꿔라.** 코어가 안 풀리면 도감 데이터·플레이버를, 그것도 막히면 휴식을. 번아웃(risk-register)이 1인 개발 최대 적.
- **데이터 주도를 믿어라.** 수치는 코드가 아니라 `data/`에 있다(tech-arch §4.4). 밸런스 조정에 코드를 안 건드린다.
- **"더"보다 "더 잘".** 컷은 패배가 아니라 출시 전략이다. 작게 내고 라이브로 키운다.
