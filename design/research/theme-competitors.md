## 2. 차별화 기회 (우리만 할 수 있는 빈 틈)

- **역방향 수치의 완전 구현**: 조사한 16개 중 숫자가 실제로 작아지는 게임은 Universe Shrinker 단 하나뿐. 그마저도 우주 스케일만 다루고 미립자 세계와 무관. **분자→원자→전자→쿼크→플랑크 8단 하강 체인을 메인 루프로 삼는 게임은 시장에 존재하지 않음.**
- **압축=자원 축적 메커닉의 부재**: 기존 게임은 모두 "생산 건물을 사서 숫자를 키운다"는 정방향. "압축 행위 자체가 에너지·자원을 만든다"는 역학은 Criticality 핵분열 체인이 부분 유사하나 크기 축소와 연결 안 됨. **압축률이 수치 생산량에 직결되는 설계는 완전 공백.**
- **입자 도감(발견 시스템)**: 16개 중 발견 → 도감 등록 → 서사 연결 구조를 가진 게임 전무. Particle Clicker가 역사적 발견 순서를 따르나 단순 업그레이드 잠금해제. **미지 입자를 발견·명명하는 행위 자체를 도파민 원천으로 만드는 설계는 비어 있음.**
- **Steam 정식작으로서 미립자 하강 테마**: Steam 출시/확정작은 Criticality(분열 방향)와 Incremental Mass Rewritten(질량 증가 방향) 둘뿐. **"작아지는 방향 + 입자 과학" 조합으로 Steam 진출하는 게임 없음.**
- **물리 스케일 계층 자체가 서사**: 기존 게임은 물리 용어를 이름표로만 사용(Quark/Atom이 그냥 통화명). 실제 크기 단위(nm, pm, fm, Planck)가 HUD에 나타나 "얼마나 깊이 들어갔는지"를 직관적으로 느끼게 하는 시각화는 공백.
- **8단 체인 오토메이션 + 물리 실재 연결**: Particle Incremental(xnoobspeakable)이 유사 계층 구조를 가졌으나 입자 계층을 물리적 실재와 연결하지 않음. 각 계층이 실제 물리 스케일과 대응되며 자동화 체인을 구성하는 설계는 없음.

## 3. 배울 점 / 피할 함정

### 배울 점

- **Criticality의 체인리액션 사운드 피드백**: 음향이 반복 클릭의 지루함을 덮고 만족감을 만든다는 것이 증명됨(88% 긍정 핵심 원인). 압축 이벤트마다 층위별 청각 피드백을 다르게 설계.
- **Fundamental의 스테이지 전환 설계**: 각 스테이지마다 메커닉이 완전히 재구성됨이 호평("계속 달라진다"). 8개 스케일 층위가 단순 반복이 아닌 새 메커닉 도입 기회여야 함.
- **Elemental Incremental의 Gas Container**: 온도/압력이 소프트캡 역할을 하며 새 수치 차원을 추가. 압축률이 열 방출·엔트로피 지표로 연결되는 설계에 참고.
- **Particle Idle의 이머전트 시각화**: 물리 시뮬로 생성되는 패턴(해파리 형태)이 커뮤니티 화제. 도감에 수집된 입자가 시뮬레이션 비주얼로 등장하는 방향이 유효.
- **Universe Shrinker의 목표 명시성**: "1 Planck Length 도달"이라는 구체적 완료 조건이 단기 목표로 작동. 각 층위마다 "몇 m까지 도달"을 명시하면 진행감 강화.

### 피할 함정

- **Universe Shrinker의 "5레이어 동안 새 것 없음"**: 레이어 반복 구조에서 메커닉 다변화 없이 숫자만 키우면 이탈. 경쟁자 공통 최대 실패.
- **Incremental Mass Rewritten의 불투명한 공식**: "업그레이드가 무엇을 어떻게 바꾸는지 알 수 없다"는 리뷰 다수. 압축 공식과 생산 증가 경로를 플레이어에게 명시적으로 표시.
- **Atom Idle의 "작은 것→큰 것" 방향 전환 유혹**: 원자 생성이 자연스럽게 블랙홀/암흑물질 상향으로 흐름. 우리 게임은 스케일 방향성을 일관되게 하향 유지하는 설계 원칙 필요.
- **Particle Clicker의 단선형 위험**: 프레스티지 없이 단선 진행 시 후반 이탈 빠름. 8단 체인 각각에 마이크로 프레스티지를 배치.
- **테마를 이름표로만 쓰는 함정**: Quark Incremental, Accelerator Incremental이 물리 용어를 통화명에만 붙여 커뮤니티에서 공허하다는 평. 각 입자의 실제 특성(질량/스핀/수명)이 게임 수치에 반영되어야 함.
- **모바일 성능 문제(Particle Idle 사례)**: 물리 시뮬 비주얼을 JS로 구현 시 모바일 프레임 드롭. 비주얼 레이어와 수치 레이어를 분리.
