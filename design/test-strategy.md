## 0. 이 문서를 읽는 법

"테스트 전략"은 어떤 버그가 치명적인지를 먼저 정하고, 치명적인 것부터 자동으로 잡는 체계를 만드는 것이다. 모든 것을 테스트할 수 없으므로 **우선순위**가 핵심이다.

이 게임에서 "치명적인 버그"의 기준은 명확하다:

1. **세이브 손상** — 방치형 최악. 수 주 진행 증발 (risk PR5).
2. **경제 수식 오류** — 페이싱 붕괴. 게임이 막히거나 폭주 (risk PR4 / R1~R9).
3. **BigNumber 경계 오류** — 게임 수치 오버플로 또는 native 연산 혼입 (risk R2).
4. **오프라인 계산 오류** — 방치형의 핵심 보상 루프가 잘못 지급됨 (risk R4).

이 순서가 테스트 우선순위의 척추다. 비주얼·UX 버그보다 위 4가지가 항상 먼저다.

**이 문서가 담당하는 레인: "코드 버그를 잡는 엔지니어 테스트"**  
5개 QA 레인(§5 참조)에서 *엔지니어 테스트*가 담당하는 영역만 다룬다. 밸런스 수치, 사람 플레이테스트, 시각 품질, 텔레메트리는 각각 담당 레인이 있다.

## 1. 테스트 피라미드

```
        ▲ e2e (소수)
       / \  — 세이브 왕복·로드·마이그레이션 시나리오
      /   \    실제 브라우저 / NW.js 환경
     /─────\
    /통합 (중간)\
   /             \ — 모듈 간 연결: 게임 루프 × 세이브 × 오프라인
  /───────────────\   이벤트 버스·StorageAdapter 주입·플랫폼 어댑터 전환
 /   단위 (다수)    \
/─────────────────────\
— 순수 함수: 수식·BigNumber·파싱·마이그레이션·직렬화
```

### 1.1 단위 테스트 (Unit)

**대상**: 외부 의존성 없는 순수 함수. 입력→출력이 결정적인 것.

- 경제 수식 (`production_mult`, `cost_k`, `QF_gain`, `dec = α·log₁₀(C)`, 오프라인 계산 공식)
- BigNumber 경계 (`Decimal` 연산, `.toString()`/`new Decimal(str)` 왕복, 특수값 방어)
- 세이브 직렬화·역직렬화 (JSON + lz-string, Decimal 변환, 봉투 구조)
- 마이그레이션 함수 각각 (v→v+1 한 칸씩, 단독 실행 가능해야 함)
- format 모듈 (과학표기·극소 표기·접두사 표기)
- 탬퍼 클램프 계산 (elapsed, capped, effective_s)

**특성**: 빠름 (수 밀리초), 수십~수백 개, 외부 의존 없음(mocking 최소), CI마다 전체 실행.

### 1.2 통합 테스트 (Integration)

**대상**: 모듈 두 개 이상이 맞물리는 지점. 실제 연결의 계약을 검증.

- 게임 루프 × 상태: N틱 실행 후 자원이 기대값 범위 안에 드는지
- 오프라인 계산 × 세이브: 저장 후 `elapsed` 계산 → 오프라인 자원 지급 → 상태 정합
- StorageAdapter × 세이브 시스템: LocalStorageAdapter로 저장 후 로드, 왕복 검증
- 이벤트 버스 × 메커니즘 모듈: 상전이 이벤트 발행 시 QF·codex·사운드 훅이 각각 수신되는지
- 마이그레이션 체인: v0 세이브 → 현재 버전까지 체인 적용 후 유효 상태가 되는지

**특성**: 단위보다 느림 (수십~수백 밀리초). CI마다 실행하되 단위 이후 순서.

### 1.3 e2e 테스트 (End-to-End)

**대상**: 사용자 시나리오 전체 흐름. 실제 런타임에서 주요 경로만.

- 세이브 왕복 전체: 게임 시작 → N틱 → 저장 → 로드 → 상태 일치
- 마이그레이션 전체: 실제 구버전 세이브 파일을 현재 버전으로 로드
- 오프라인 복귀 시나리오: 저장된 `lastSave`를 X시간 과거로 설정 → 로드 → 오프라인 보너스 지급 확인
- 첫 상전이 달성: 시뮬 속도로 dec19까지 진행 → QF 획득 → `production_mult` 상승 확인

**특성**: 느림 (초 단위). 개수는 최소로 (핵심 경로만). CI에서는 실행하되 실패 시 "느린 테스트 게이트"로 분리해 빠른 피드백 차단 안 함.

## 2. 단위 테스트 우선순위 — tech-arch R1~R9 연계

**tech-arch §7의 R1~R9 리스크를 "어떤 테스트가 잡는가"로 연결**한다. 리스크 순서 = 테스트 작성 우선순위 순서.

### 우선순위 1 — 세이브·마이그레이션 (R1·R5·R7)

세이브 관련 버그는 발견이 늦을수록 피해가 누적된다. **F-게이트(1일차)와 함께 가장 먼저 작성**.

| 테스트 대상 | 무엇을 잡는가 | 리스크 |
|---|---|---|
| **봉투 포맷 파싱** (`{version, data, checksum}`) | `version`이 없는 구버전 봉투, 필드 누락, 비정상 JSON | R1·R5 |
| **체크섬 검증** | `data`가 변조되면 체크섬 불일치 감지 | R5 |
| **마이그레이션 함수 각각** | v→v+1 변환이 유효 상태를 만드는지. 새 마이그레이션 추가 시 반드시 테스트 함께 | R1 |
| **마이그레이션 체인** | v0(봉투 이전) → 현재까지 순차 적용. 처음부터 끝까지 한 번에 통과 | R1 |
| **`validate` 기본값 채움** | 누락 필드에 올바른 기본값이 주입되는지 | R7 |
| **Decimal 직렬화 왕복** | `.toString()` → `new Decimal(str)` 왕복이 정밀도 손실 없이 복원되는지. 큰 값(1e308 급)·작은 값(1e-35 급) 모두 | R1 |
| **특수값 방어** | `Infinity`, `NaN`, `null`이 세이브에 들어오면 안전하게 처리(catch·기본값)되는지 | R5 |
| **lz-string 인코딩 일관성** | `compressToUTF16`/`compressToBase64` 각각이 자기 복호화와 일치하는지. 혼용 시 오류가 발생하는지 | R1 |
| **`.corrupt.bak` 보존** | 마이그레이션 실패 시 원본이 삭제되지 않고 보존되는지 (StorageAdapter mock) | R5 |

### 우선순위 2 — BigNumber 경계 (R2)

native 연산이 슬며시 섞이면 재하강 후반에서 오버플로가 터진다. **구현 직후 작성**.

| 테스트 대상 | 무엇을 잡는가 | 리스크 |
|---|---|---|
| **핵심 수식이 `Decimal` 반환하는지** | `production_mult`, `cost_k(n)`, `QF_gain` 등 반환 타입 검증 | R2 |
| **`lifetime_C` 누적** | 재하강을 반복하며 `lifetime_C`가 단조 증가하는지. 오버플로 시 break_eternity가 올바른 표현을 쓰는지 | R2 |
| **`bought`(native int) → 비용(Decimal)** | `bought`가 큰 정수(수천)일 때도 `cost_k(bought)` 계산이 Decimal로 올바르게 나오는지 | R2 |
| **성능·정밀도 경계** | 극값(1e1e308, 1e-(1e308))에서 `Decimal` 연산이 throw하지 않고 표현되는지 | R2 |

### 우선순위 3 — 오프라인 계산·탬퍼 클램프 (R3·R4·R9)

오프라인은 방치형의 핵심 보상이므로 수식 오류가 게임플레이를 직접 망가뜨린다.

| 테스트 대상 | 무엇을 잡는가 | 리스크 |
|---|---|---|
| **기본 일괄 계산** | `elapsed`, `capped`, `effective_s` 각각이 economy §3.1 공식과 수치 일치 | R4·R9 |
| **탬퍼 클램프(48h)** | `elapsed = 100h` 입력 시 `effective_s`가 48h 기준 값을 초과하지 않는지 | R4 |
| **음수 elapsed 방어** | `now < lastSave` (시계 역행)이면 elapsed=0, 자원 지급 없음 | R4 |
| **상전이 직후 modifier=1.0** | 플래그가 켜진 상태에서 오프라인 계산 시 modifier=1.0 적용 후 플래그 해제 | R4 |
| **오프라인 적용 후 자원 정합** | `E`·`C` 가산 결과가 기대 범위 안에 드는지 (economy §3 시뮬 출력과 비교) | R9 |
| **CAP(24h) 적용** | `elapsed = 30h` → `capped = 24h`인지 | R3·R4 |
| **long_idle 배율** | `elapsed > 24h`일 때 `long_idle = 1 + LB·log10(...)` 공식이 economy §3.1과 일치 | R4 |

### 우선순위 4 — economy 수식 핵심 (시뮬 검증값 일치)

economy.md §1~§2의 확정 수식이 코드로 올바르게 구현되는지. **qa-balancer 시뮬과 같은 입력→같은 출력**이어야 회귀를 잡을 수 있다.

| 테스트 대상 | 기대값 (economy.md 출처) | 목적 |
|---|---|---|
| **`QF_total` 수식** | `floor(K·(lifetime_C/D_norm)^0.5)`, K=1, D_norm=1e26 | 공식 오타·지수 오류 감지 |
| **`QF_gain`** | `QF_total - QF_claimed`, 음수 방어(0 클램프) | 중복 청구 방지 |
| **`production_mult`** | `1 + 0.25·log10(1+QF_total)`. QF=41 → mult≈1.406 (economy §1.2 PT2 입력값) | economy §1.2 표 재현 |
| **`dec` 계산** | `α·log10(C)`. α 파라미터 고정값 검증 | dec 수식 오타 방지 |
| **`cost_k(n)`** | `base_k·growth_k^n`. growth_k=2.2→1.8 내림차순(economy §2) | 비용 곡선 회귀 |
| **벽 도달 판정** | `dec >= wall_value`일 때 상전이 트리거 | WALLS=[19,21.5,...,26] 하드코딩 회귀 |
| **`production_mult` 합산** | research_mult, codex_bonus, holographic_mult 결합 시 economy §7.2.6 수치 범위 안 | 배율 스택 버그 |

**시뮬 검증값 활용**: economy.md §1.2 표의 `런 시간`, `QF 누적` 등은 테스트의 "골든 마스터값"이다. 단, **이 값들은 능동 플레이 가정(dt=1~2s) 시뮬 출력**이므로, 테스트에서는 동일 dt와 초기 상태로 N스텝 실행 후 오차 허용 범위(±5%)로 비교한다.

## 3. 회귀 전략

### 3.1 세이브 호환성 회귀

**세이브 마이그레이션 함수는 한 번 작성 후 영원히 보존하고, 그 테스트도 영원히 보존한다** (tech-arch §1.3 규칙).

- **픽스처(fixture) 기반**: `tests/fixtures/` 에 실제 게임에서 뽑은 각 schema_version의 세이브 스냅샷을 보관. 새 버전이 추가될 때마다 해당 버전 픽스처도 추가.
- **회귀 테스트**: 각 픽스처를 현재 버전으로 로드 → 유효 상태 확인. 마이그레이션 체인이 추가될 때마다 전 픽스처에 대해 재실행.
- **"구 세이브를 버린다" = 절대 금지 신호**: 픽스처 하나라도 현재 버전에서 로드 실패하면 **배포 차단**. 세이브 호환성은 흥정 대상이 아니다.

### 3.2 밸런스 회귀 (골든 마스터)

qa-balancer 시뮬과의 연결 지점.

- **골든 마스터 정의**: economy.md §1.2가 확정한 수치(첫 상전이 4.26h, PT2 진입 mult 1.406, 빅 크런치 진입 mult 2.759 등)가 "골든 마스터 테이블"이다.
- **테스트 방법**: 게임 루프를 "fast-forward 모드"(dt를 실시간이 아니라 고정 1s, rAF 없이 루프)로 N스텝 실행 → 특정 dec 도달 시각을 측정 → 골든 마스터와 ±5% 비교.
- **`data/` 파라미터 변경 감지**: `data/chain.json`(base_k·growth_k 등) 또는 `data/walls.json`이 변경되면 골든 마스터 테스트가 실패할 수 있다. 이것은 의도된 설계 변경인지 실수인지를 구분하게 해주는 안전장치. **의도적 변경 → 골든 마스터 업데이트 + 주석으로 근거 기록.**
- **qa-balancer 시뮬과 차이**: qa-balancer의 python 시뮬은 Euler dt=1~2s의 **연속 시뮬**이고, 게임 엔진 테스트는 **이산 TypeScript 구현**이다. 미세 오차(±5%)는 허용하되 계통 오차(10% 이상, 특히 단조 증가 방향 위반)는 버그로 처리.

### 3.3 BigNumber 직렬화 회귀

- break_eternity의 `.toString()` 포맷이 라이브러리 업데이트로 바뀌면 기존 세이브가 깨진다 (tech-arch R1). 의존성 업데이트 시마다 "Decimal 왕복 테스트"(§2 우선순위 2)를 반드시 실행.
- **의존성 잠금**: `pnpm-lock.yaml`을 항상 커밋. break_eternity 버전은 major 고정(`"break_eternity.js": "~1.x.x"`). 업그레이드는 수동·의도적으로만.

## 4. 도구 방향

### 4.1 Vitest — 주 테스트 프레임워크

| 선택 | 근거 |
|---|---|
| **Vitest** | Vite·TypeScript 생태계와 네이티브 통합. ESM 모듈 변환 없이 Svelte·TS import 그대로 테스트. Jest 호환 API(jest→vitest 마이그레이션 비용 최소). 단위·통합 테스트 모두 담당. |

- **실행 모드**: `vitest run`(CI 1회), `vitest`(개발 watch 모드), `vitest --coverage`(커버리지).
- **설정**: `vite.config.ts`에 통합(별도 설정 파일 최소화). `test.environment`는 단위는 `node`, DOM이 필요한 UI 통합은 `jsdom`.

### 4.2 테스트 파일 위치 규칙

```
src/
  core/
    save/
      save.ts
      save.test.ts          ← 단위 테스트 (같은 폴더)
    chain/
      chain.ts
      chain.test.ts
  tests/
    integration/            ← 통합 테스트 (모듈 간 연결)
      save-loop.test.ts
      offline-state.test.ts
    e2e/                    ← e2e (전체 시나리오)
      save-roundtrip.test.ts
    fixtures/               ← 세이브 픽스처 (schema별)
      save_v0.json
      save_v1.json
    golden/                 ← 밸런스 골든 마스터 테이블
      economy-milestones.json
```

- **단위 테스트는 해당 파일 옆에**: 파일을 이동하면 테스트도 따라간다.
- **통합·e2e·픽스처는 `tests/`로 분리**: 여러 모듈에 걸쳐 있거나 외부 파일을 참조하는 것.

### 4.3 자동 vs 수동

| 검사 종류 | 자동 (CI) | 수동 |
|---|---|---|
| 단위 테스트 | 항상 (모든 push/PR) | — |
| 통합 테스트 | 항상 | — |
| e2e 테스트 | CI (별도 느린 게이트) | 릴리스 전 수동 재확인 |
| 세이브 픽스처 회귀 | 항상 | 릴리스 전 실제 빌드에서도 확인 |
| 골든 마스터 | 항상 (`data/` 변경 PR에서) | `data/` 대규모 변경 시 qa-balancer 시뮬 재실행 |
| 퍼포먼스 테스트 (프레임·메모리) | 없음 (수동) | 정거장 2·3 마일스톤마다 |
| 시각 품질 | 없음 (visual-qa 레인) | — |
| 실플레이어 테스트 | 없음 (playtest-plan 레인) | — |

## 5. ★ 5개 QA 레인 통합 지도

이 게임의 품질 보증은 5개 레인으로 나뉜다. 각 레인은 **다른 것을 잡는다** — 겹치는 것처럼 보이지만 실제로는 다른 계층의 버그다.

```
          무엇을 잡는가?

코드 버그  ┌─────────────────────────────────────────────────────┐
          │ 1. 엔지니어 테스트 (본 문서)                          │
          │    단위·통합·e2e. 수식 오류·세이브 손상·             │
          │    Decimal 경계·마이그레이션 실패.                    │
          │    "코드가 설계대로 돌아가는가"                       │
          └─────────────────────────────────────────────────────┘
                          ↕ (골든 마스터·수식 공유)
밸런스 수치 ┌─────────────────────────────────────────────────────┐
          │ 2. qa-balancer (python 시뮬)                        │
          │    페이싱·진행 막힘·폭주·프레스티지 타이밍           │
          │    자동 플레이 시뮬. "수치가 의도대로 작동하는가"      │
          └─────────────────────────────────────────────────────┘
                          ↕ (골든 마스터가 2→1로 검증값 흘림)
사람 경험   ┌─────────────────────────────────────────────────────┐
          │ 3. playtest-plan (실플레이어)                       │
          │    재미·FTUE·동기·리텐션 주관.                       │
          │    "사람이 실제로 즐기는가"                           │
          └─────────────────────────────────────────────────────┘

시각 품질   ┌─────────────────────────────────────────────────────┐
          │ 4. visual-qa                                        │
          │    스크린샷·해상도·DPI·애니메이션·주스                │
          │    "보기에 맞는가"                                   │
          └─────────────────────────────────────────────────────┘

실사용 데이터 ┌───────────────────────────────────────────────────┐
          │ 5. analytics-plan (텔레메트리)                      │
          │    D1/D7/D30 리텐션·퍼널·D_total 분포.              │
          │    "실플레이어가 어디서 그만두는가" → R8 재튜닝       │
          └─────────────────────────────────────────────────────┘
```

### 5.1 겹치는 지점과 경계

| 겹침처럼 보이는 것 | 실제 담당 레인 | 경계 |
|---|---|---|
| "첫 상전이 4.26h" | qa-balancer가 **설계 목표로 확정** → 엔지니어 테스트가 **구현 일치 검증** | 수치 변경권은 economy/qa-balancer. 엔지니어 테스트는 그 수치를 골든 마스터로 받아 코드가 일치하는지만 잡음. |
| "세이브 손상" | 엔지니어 테스트(코드 버그) + analytics-plan(실측 빈도) | 엔지니어 테스트는 로직 오류를 사전에 잡음. 텔레메트리는 출시 후 실제 발생 빈도를 모니터링. |
| "밸런스 수치가 틀리다" | qa-balancer가 **설계 차원** 잡음 / 엔지니어 테스트가 **구현 차원** 잡음 | qa-balancer: "economy 공식의 파라미터가 맞는가". 엔지니어: "그 공식이 코드로 올바르게 구현되었는가". |
| "재미없다" | playtest-plan만 | 엔지니어 테스트는 재미를 측정하지 않는다. "코드가 돌아가는가"와 "재미있는가"는 다른 질문. |
| "D_total 분포 이상" | analytics-plan이 EA 실데이터로 감지 → economy가 재튜닝 → 엔지니어 테스트가 재튜닝 반영 검증 | R8 파이프라인. 엔지니어 테스트는 재튜닝 후 골든 마스터 업데이트 + 구현 일치 재확인 역할. |

### 5.2 레인 간 협력 프로토콜

- **qa-balancer → 엔지니어 테스트**: economy.md가 확정한 수치 변경 시 골든 마스터 업데이트. 엔지니어가 `tests/golden/economy-milestones.json`을 갱신하고 테스트 통과를 확인한 뒤 merge.
- **엔지니어 테스트 → qa-balancer**: 구현에서 수식 오타·경계 오류를 발견하면 economy/systems에 보고. 임의로 수식을 수정하지 않는다(game-programmer 원칙: "설계를 충실히 구현, 임의 변경 금지").
- **엔지니어 테스트 → playtest-plan**: 세이브 데이터 픽스처가 플레이테스트 세션의 "초기 상태 주입"에 활용될 수 있음(특정 dec에서 시작하는 테스트 세이브 제공).
- **analytics-plan → 엔지니어 테스트**: D_total p99 알람이 경보 임계를 넘으면 → economy 재튜닝 → 코드 변경 → 골든 마스터 업데이트 흐름. 텔레메트리 이벤트 발행 코드 자체는 엔지니어 테스트 범위(이벤트가 올바른 시점에 올바른 값으로 발행되는지).

## 6. 커버리지 목표 (현실적)

**커버리지 숫자 자체는 목표가 아니다.** "커버리지 80%"를 달성해도 수식 경계값이 안 테스트되면 의미 없다. 커버리지는 **중요한 경로가 빠졌는지 체크하는 도구**로만 쓴다.

| 영역 | 목표 커버리지 | 근거 |
|---|---|---|
| `core/save/` (직렬화·마이그레이션·봉투) | **90%+** | PR5(세이브 손상) = 치명. 한 줄도 빠지면 안 되는 코드. |
| `core/bignum/` (Decimal 래퍼·가드) | **90%+** | R2 BigNumber 경계. 모든 경로에 Decimal 보장 확인. |
| `core/offline/` (오프라인 계산·탬퍼 클램프) | **85%+** | R4 탬퍼 클램프 = 핵심 방어선. 공식 분기 전부 커버. |
| `core/chain/` (8단 체인 비용·생산) | **85%+** | economy 수식 구현. 골든 마스터와 일치하는 경로. |
| `core/prestige/` (QF·상전이·재하강) | **80%+** | QF 수식·modifier 플래그. 핵심 프레스티지 루프. |
| `core/format/` (표시 변환) | **75%+** | 오류가 게임플레이에 영향 없음. 그러나 극소 표기는 커버. |
| `ui/` (Svelte 컴포넌트) | **50%+** | 렌더는 로직 없음(§4.1 분리 원칙). e2e에서 간접 검증. |
| `render/` (파티클·VFX) | **측정만** | graphics-programmer 영역. 코드 버그보다 visual-qa가 잡음. |

## 7. 단계별 테스트 작성 계획

### 정거장 1 (F-게이트와 함께)

**F-게이트(세이브·Decimal·루프) 구현 시 테스트를 동시에 작성한다.** 테스트 없는 F-게이트는 F-게이트가 아니다.

| 마일스톤 | 작성할 테스트 |
|---|---|
| **M1.1** (F-게이트 + 빈 루프) | 세이브 봉투 파싱, Decimal 왕복, lz-string 왕복, 체크섬 검증 |
| **M1.2** (8단 체인 + 수식) | `cost_k`, `production_mult`, `dec` 계산, 골든 마스터 초안 |
| **M1.3** (온보딩) | 벽 도달 판정, 층 진입 로직 |
| **M1.4** (능동 손맛) | 클릭 생산 계산 (메커니즘 단위) |
| **M1.5** (첫 상전이) | `QF_gain`, 상전이 직후 플래그, modifier=1.0 플래그 |
| **M1.7** (오프라인) | 탬퍼 클램프, elapsed 계산, 음수 방어, CAP 적용 |

### 정거장 2~3 (확장)

- 마이그레이션 픽스처: 각 schema_version 세이브 추가(세이브 포맷이 바뀔 때마다).
- 통합 테스트: StorageAdapter 왕복, 이벤트 버스 연결.
- 골든 마스터 완성: economy §1.2 표 전체를 `economy-milestones.json`으로.
- 텔레메트리 이벤트 발행 테스트 (M3.8과 함께).

## 8. 요약 — 4줄

1. **테스트 피라미드**: 단위(순수 함수·수식·직렬화) 다수 → 통합(모듈 연결·StorageAdapter) 중간 → e2e(세이브 왕복·마이그레이션 전체 경로) 소수. 도구는 Vitest(Vite·TS 네이티브).
2. **우선순위 순서**: 세이브·마이그레이션(R1·R5·R7) → BigNumber 경계(R2) → 오프라인·탬퍼 클램프(R4) → economy 수식(R9 골든 마스터). "치명적인 것 먼저"가 원칙.
3. **회귀 전략**: 세이브 픽스처(schema 버전별 영구 보존) + 골든 마스터(economy.md §1.2 확정값) + Decimal 라이브러리 버전 잠금. 픽스처 로드 실패 = 배포 차단.
4. **5개 QA 레인**: 엔지니어(코드 버그) ↔ qa-balancer(밸런스 수치) ↔ playtest(사람 재미) ↔ visual-qa(시각) ↔ analytics(리텐션·D_total 실측). 각 레인이 다른 계층을 잡고, 골든 마스터가 qa-balancer→엔지니어로 검증값을 공유한다.
