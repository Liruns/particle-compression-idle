## 1. 왜 이 게임에 텔레메트리가 필요한가 (설계 관점)

이 게임의 **설계 리스크는 낮다** — 코어 수식·6벽·재하강이 시뮬·독립 QA·디렉터 R3로 닫혔다(risk §0). 그러나 **두 가지는 시뮬로 닫을 수 없고 오직 실플레이어 데이터로만 닫힌다**:

- **(A) D_total magnitude** — 가장 큰 잔여 밸런스 리스크(risk R8, scope C4, director R3 동일 지목). holo_factor=0.008이 D_total≲1e7을 가정하는데, **D 생산률이 미확정**(research-tree 노드별 D 비용 TBD)이라 실제 magnitude를 모른다. 시뮬은 가정만 검증; 실측이 필요.
- **(B) 리텐션·권태 지점** — 첫 빅 크런치 0.82주 = "짧은 게임" 인식 리스크(PR4-a), 재하강 2회차+ 권태(PR4-b). 디렉터는 *설계 측면*을 닫았으나(§8.1) **실플레이어 반응은 미검증**.

EA 채택의 핵심 근거가 바로 이것이다(launch §4-B #3): EA = 실데이터로 밸런스를 정직하게 보정하는 채널. **그러므로 텔레메트리 *수집* 코드는 v1.0 출시 빌드에 반드시 들어간다** — 안 넣으면 출시 후에도 데이터가 없어 영영 보정 못 한다(scope §5-1 판정 B). 분석·대시보드 자체는 데이터가 쌓인 뒤(출시 후)지만, **수집은 1일차부터**.

> **설계 원칙**: 모든 목표 분포는 economy 시뮬이 실증한 페이싱에서 도출한다. 텔레메트리는 "설계가 맞았는지"를 검증하는 *대조군*이지, 새 설계가 아니다. 실측이 목표 밴드를 벗어나면 → economy 재튜닝 트리거(어느 파라미터를, 어느 방향으로 조정할지까지 §6에 명시).

## 2. 리텐션 지표 (D1 / D7 / D30) ★

### 2.1 방치형 특성에 맞는 리텐션 정의

표준 "다음날 재접속" 정의를 쓰되, **이 게임의 페이싱 마일스톤에 앵커링**한다. 방치형은 "매일 길게"가 아니라 "며칠에 걸쳐 짧게 자주 + 오프라인이 일함"이므로, 리텐션 윈도우를 캠페인 비트에 맞춘다.

| 지표 | 정의 | 이 게임에서의 의미 (economy 앵커) |
|---|---|---|
| **D1** | 설치 후 24~48h 내 1회+ 세션 | Hook→Habit 진입. PT1(첫 상전이)이 캘린더 ~0.59일(기본 플레이어)에 도달하므로, **D1은 "첫 상전이 도파민을 맛본 뒤 돌아오는가"**를 측정. |
| **D7** ⭐ | 설치 후 6~8일 내 1회+ 세션 | **키스톤.** 첫 빅 크런치가 **5.77일(0.82주, economy §4.2)**에 떨어진다 → D7이 정확히 **"1회차 캠페인 클라이맥스(첫 대발견) 도달"**을 측정. 이 게임 리텐션의 단일 최중요 지표. |
| **D30** | 설치 후 28~32일 내 1회+ 세션 | 재하강 루프 점착. 재하강 ×3 = 2.47주(economy §4.4)이므로 D30 잔존자는 **~4~5회차 재하강 깊이**. "수 주 리텐션"(멀티위크) 실현 여부. |

- **윈도우에 ±1~2일 여유**: 방치형 플레이어는 매일 안 들어와도 정상(오프라인이 진행) → 엄격한 "정확히 그날"이 아니라 윈도우로 측정. D1=[24h,48h), D7=[6일,8일), D30=[28일,32일).
- **세션 정의**: 앱 포그라운드 진입 + 1회+ 의미있는 액션(구매/상전이/체크인). 단순 실행 후 즉시 종료(<10초)는 세션 제외(노이즈).

### 2.2 리텐션 목표 (구조 도출 — 미검증)

> ⚠️ **전부 미검증.** 아래는 economy 페이싱 + 방치형 업계 벤치마크에서 도출한 *구조 목표*다. 실측 전엔 가정. 출시 후 첫 코호트로 교체.

| 지표 | 업계 방치형 벤치 | **본 게임 구조 목표** | 근거 |
|---|---|---|---|
| D1 | ~30~40% | **≥35%** | 온보딩 1.2h + PT1(0.59일) 도파민이 day1 재방문을 끌어야. |
| **D7** | ~12~20% | **≥15%** | 첫 빅 크런치(0.82주) 도달 = 멀티위크 약속의 게이트. PR4-a 직접 측정. |
| D30 | ~4~8% | **≥6%** | 재하강 루프가 D7의 ~40%를 장기 전환해야. |
| D7/D1 | — | **≥0.40** | Habit 전환율. 0.40 미만 = 첫 주 페이싱이 지루(PR4). |
| **D30/D7** | — | **≥0.40** | **재하강 루프 점착.** 0.40 미만 = 2회차+ 권태(PR4-b)의 직접 신호. |

- **D30/D7가 PR4-b(재하강 권태)의 핵심 게이지**: 재하강이 "또 돌게" 만드는지를 D7→D30 생존으로 측정. 이 비율이 낮으면 회전 집중 서브층(systems)이 새로움을 못 주고 있다는 뜻 → 후기 회차 변별(scope C2) 우선순위 상승.
- **세그먼트별 리텐션**: 인게이지먼트 스타일(세션/일 빈도)로 코호트 분리. economy §4.3이 캐주얼(1회/일)~인게이지드(3회/일)로 첫 크런치 0.78~0.86주 수렴을 예측 → **리텐션도 스타일 무관하게 수렴해야 정상**(방치 보상 필러 ③ 검증). 캐주얼 코호트 D7이 인게이지드보다 크게 낮으면 = 방치 경로가 보상받지 못함(필러 위반).

### 2.3 방치형 특화 리텐션 보조 지표

표준 DN만으로는 "방치형이 잘 돌고 있는지" 부족하다. 보조로:

- **오프라인 복귀율**: 오프라인 크레딧 수령 화면을 본 세션 / 전체 세션. 방치→복귀 루프가 도는지. (낮으면 "닫았다 켜는 쾌감"이 약함 = P3 방치측 실패.)
- **세션 간격 분포**: 체크인 케이던스 실측. economy는 2회/일(14h/10h 갭)을 가정 → 실제 분포가 이와 다르면 오프라인 캡(24h)·modifier(0.65) 재검토 입력.
- **장기 휴면 후 복귀(resurrection)**: 7일+ 미접속 후 재접속. 오프라인 캡 24h가 "1주 비웠다 와도 보상"을 충분히 주는지(economy §3.2: 168h 방치 = 17.95h 크레딧, 11%). 부활률이 낮으면 장기 방치 보상 재검토.

## 3. 퍼널 / 이탈 지표 — "어디서 그만두나" ★

방치형 이탈은 **특정 페이싱 벽에서 발생**한다. 7상전이·6서브층·재하강 회차 경계마다 도달율을 측정해 이탈 지점을 핀포인트한다.

### 3.1 온보딩 퍼널 (첫 세션 ~ PT1)

가장 이탈이 큰 구간. 알려진 물리(dec0~9, 1.2h) → 첫 상전이(PT1, dec19).

| # | 퍼널 단계 | 이벤트 | 무엇을 잡나 |
|---|---|---|---|
| F0 | 게임 시작 | `game_start` | 분모(설치→첫 실행) |
| F1 | 첫 압축기 구매 | `first_compressor_bought` | 첫 능동 액션. dec2~3 험프(13분+34분)에서 이탈하는지(economy §2.2) |
| F2 | 첫 자동화 | `first_automation` | 능동→방치 전환점 통과 |
| F3 | 첫 층 진입(무상전이) | `layer_enter` (known physics) | "층 진입 ≠ 상전이" 학습 성공(P2·P5) |
| F4 | 쿼크 한계 도달 | `dec_milestone` (dec9) | 알려진 물리 완료(온보딩 졸업) |
| F5 | **PT1 도달** | `phase_transition` (idx=1) | **온보딩 퍼널의 골.** 첫 상전이 = 첫 QF. 4~6h 페이싱이 너무 길어 이탈하는지 |

- **온보딩 완료율** = F5 도달 / F0. 이 게임에서 가장 중요한 단일 퍼널 수. PT1까지 능동 4.26h(캘린더 0.59일)는 *방치형치고 긴 첫 보상*이므로 F2~F5 사이 이탈을 면밀히. (F2 자동화 전 험프에서 이탈하면 = 온보딩이 능동 부담 과다.)
- **드롭오프 핫스팟 자동 식별**: 각 F단계 간 이탈률을 보고 가장 큰 낙폭 구간을 플래그. 설계 예상(F1 험프, F5 첫 상전이 대기)과 실제가 다르면 FTUE·페이싱 재조정.

### 3.2 코어 진행 퍼널 (PT1 ~ 첫 빅 크런치)

6서브층 × 상전이. **각 서브층/상전이별 도달율 + 체류시간**으로 "어느 층에서 그만두나"를 측정.

| 단계 | 이벤트 | 서브층 | 벽(dec) | 설계 신영역 체류 (economy §1.2a) |
|---|---|---|---|---|
| PT1 | `phase_transition` idx=1 | 프리온 | 19 | 4.26h |
| PT2 | `phase_transition` idx=2 | 끈 | 21.5 | 4.14h |
| PT3 | `phase_transition` idx=3 | 루프 | 23 | 4.23h |
| PT4 | `phase_transition` idx=4 | 거품 | 24.5 | 7.04h |
| PT5 | `phase_transition` idx=5 | 정보 | 25.5 | 7.03h |
| PT6 | `phase_transition` idx=6 | 플랑크 | 26 | 4.48h |
| **PT7** | `big_crunch` (run_index 증가) | 빅 크런치 | 26 | 첫 캠페인 완주 |

- **상전이별 도달율(survival curve)**: PT_k 도달 / PT1 도달. 단조 감소가 정상이나, **특정 PT에서 급락**하면 그 서브층이 "권태 벽". 설계상 PT4·PT5 체류가 7h로 가장 길다(거품·정보) → 여기서 이탈 집중 예상 → 우선 감시.
- **서브층 체류시간 분포**: 각 서브층의 실제 신영역 체류(능동+방치 캘린더). 설계 ≥4h 게이트(FAIL-1 해소)가 실제로 지켜지는지. 만약 어떤 층 체류가 설계값과 크게 다르면(예: 부스트로 9분 재발) = 벽 재튜닝(economy §1.2a 회귀).
- **빅 크런치 도달율** = PT7 / 설치. PR4-a("짧은 게임 인식")의 정량 — 0.82주에 첫 크런치 도달하는 코호트 비율. 이게 D7과 강하게 상관해야 정상.

### 3.3 재하강 퍼널 (멀티위크 리텐션)

| 단계 | 이벤트 | 누적 캘린더(방치, economy §4.4) | 무엇을 잡나 |
|---|---|---|---|
| 재하강 1회차 | `big_crunch` run_index=1 | 5.77일(0.82주) | 첫 크런치 |
| 재하강 2회차 | `big_crunch` run_index=2 | 11.5일(1.65주) | "또 돌았다" — 루프 성립 |
| 재하강 3회차 | `big_crunch` run_index=3 | 17.3일(2.47주) | "수 주" 달성 |
| 재하강 4~5회차+ | `big_crunch` run_index≥4 | 23~29일+ | 장기 점착 |

- **재하강 회차 분포(권태 게이지)** = PR4-b 직접 대응(risk PR4 조기경보 "재하강 2→3회차 리텐션 급락"). run_index 히스토그램에서 **2→3 사이 급락**이 권태 신호. 설계는 회차마다 동일 ~5.8일 캘린더지만 회전 집중 서브층으로 콘텐츠가 달라야 함(economy §7.5) → 급락 시 회전 로테이션이 새로움을 못 줌 = scope C2(후기 회차 변별) 우선순위 상승.
- **집중 서브층 선택 패턴**: 재하강 시 어느 서브층을 집중 선택하는가(systems §2-K). 특정 층에 쏠리면(예: 항상 같은 층) = 로테이션 매력 불균형. 다양하면 = 회전 차별화 성공.
- **개입/방치 경로 분리**: 재하강 2회차 캘린더가 방치(×1.00, 5.77일) vs 개입(×0.80, 4.12일)으로 갈리는지(economy §7.5.2). 둘 다 존재해야 필러 ③ 충족.

## 4. 밸런스 검증 텔레메트리 ⭐ (economy 전담 핵심)

설계 시뮬이 실증한 페이싱을 **실데이터로 대조**한다. 각 지표는 economy.md의 검증값을 *목표 분포*로 갖고, 실측이 벗어나면 재튜닝 트리거.

### 4.1 실제 첫 상전이(PT1) 시간 분포 — 설계 4~6h 검증

- **측정**: `time_to_pt1` = `game_start` → `phase_transition` idx=1. **두 가지로 분리**:
  - **능동 엔진-시간**(active engine-time): 실제 게임 로직이 진행한 시간(오프라인 크레딧 포함 유효시간). 설계 목표 = **4.26h**(economy §1.2), 밴드 **4~6h**.
  - **캘린더 시간**(wall-clock): 설치~PT1 실시간. 설계 목표 = **0.44~1.0일**(인게이지먼트 스타일별, economy §4.3). 중앙값 ~0.59일(기본).
- **목표 분포**: 능동 엔진-시간 p50 ∈ [4h, 6h]. 캘린더 p50 ~0.59일, p10~0.44일(인게이지드), p90~1.0일(캐주얼).
- **재튜닝 트리거**: 
  - 능동 p50 < 4h → 첫 상전이가 너무 빠름(온보딩 도파민 조기 소진) → 첫 벽 dec19 상향 검토.
  - 능동 p50 > 6h → 첫 보상이 너무 멀어 이탈(F5 드롭) → dec19 하향 또는 온보딩 단축.
  - **왜 능동/캘린더 분리가 중요한가**: 캘린더는 체크인 케이던스가 지배(방치형 특성)하므로, 페이싱 자체의 건강은 *능동 엔진-시간*으로 봐야 한다. 캘린더만 보면 "느린 플레이어"와 "느린 설계"를 혼동.

### 4.2 능동/방치 비율 — 60/40 룰 검증 (정밀 정의 필요)

> ⚠️ **중요한 정의 명확화**: "60/40 룰"을 **진행-시간 비율로 측정하면 안 된다.** economy §3·§4.2 기준 *진행에 기여하는 유효 초*는 오프라인 크레딧이 ~97%를 차지한다(기본 플레이어: 능동 버스트 0.5h/일 vs 오프라인 크레딧 15.6h/일, 본 문서 §8 재현). 60/40은 **진행-시간 분할이 아니라 "능동 입력의 *여지*가 있는가"라는 인게이지먼트 설계 의도**다(economy §2.2 "능동 입력 여지", scope P3).

따라서 60/40 룰을 **두 축으로 분해해 검증**한다:

- **축 1 — 방치 우위 검증(P3 방치측)**: `offline_credited_seconds` / (`offline_credited_seconds` + `session_active_seconds`). **목표: 방치-크레딧이 진행의 ≥50%인 플레이어가 ≥70%.** 방치형이 방치로 진행되는지. (이게 낮으면 = 능동을 강요하는 그라인드 = 방치형 실패.)
- **축 2 — 능동 참여 검증(P3 개입측)**: 세션 빈도(세션/일) + 능동 액션 밀도(클릭·공명·수동압축/세션). **목표: 세션 1~3회/일(캐주얼~인게이지드 밴드), 능동 액션이 0이 아닌 세션 ≥60%.** "눌러야 할 이유"가 작동하는지(scope F1 능동 손맛).
- **재튜닝 트리거**:
  - 축 1이 50% 미만 다수 → 오프라인 modifier(0.65) 상향 또는 능동 게이트 완화.
  - 축 2에서 능동 액션 0 세션이 과다 → 능동 메커니즘(오비탈 공명 등)의 보상 부족 → 능동 인센티브 강화.
- **보조: 능동 부스트 기여도**: 능동 세션이 진행을 얼마나 가속하는가(능동 세션 후 진행률 Δ vs 순수 방치). economy §4.3의 "개입은 첫 상전이를 0.44 vs 1.0일로 앞당김"이 실제로 재현되는지.

### 4.3 D_total 분포 ⭐⭐ (R8 / C4 — holo_factor 재튜닝의 단일 결정 데이터)

> **이 게임 텔레메트리의 최우선 산출.** holo_factor=0.008은 D_total≲1e7을 가정하나 **D 생산률 미확정**으로 실제 magnitude를 모른다. 이 히스토그램 하나가 R8 재튜닝을 결정.

- **측정**: `D_total` = `D_current` + `D_lifetime`(tech-arch §1.1 resources 네임스페이스). 수집 시점:
  - **(a) 첫 빅 크런치 시점** `big_crunch` run_index=1: D_total 스냅샷.
  - **(b) 각 재하강 회차** run_index별 D_total(D_lifetime 누적 추세).
  - **(c) 분위수**: 전 플레이어 p50/p90/**p99**/max.
- **목표/안전 밴드** (본 문서 §8 재현, economy §7.2.3):

  | D_total | D항(=log10(D+1)×0.008) | worst-case dec26 단축 | 판정 |
  |---|---|---|---|
  | 1e5 | +4.0% | −28.8% | OK (여유) |
  | 1e6 | +4.8% | −29.3% | OK (임박) |
  | **1e7** | **+5.6%** | **−29.9%** | **경계 (30% 직전)** |
  | 1e8 | +6.4% | −30.4% | **VIOLATION** |

- **알람 임계 (리드타임 확보)**: 
  - **p99 D_total > 1e6 → 주의 알람**(아직 OK지만 1e7 경계로 접근 중, 재튜닝 준비).
  - **p99 D_total > 1e7 → 즉시 재튜닝**(worst-case가 30% 가드레일 돌파 가능).
  - **p99 D_total < 1e5 → 헤드룸 존재**(holo_factor 상향 여지 = D항 보상 강화 가능).
- **재튜닝 방향**: p99가 1e7 초과 시 holo_factor를 **0.008 → log 역산으로 하향**(D_total_p99에서 D항 ≤+5.6% 유지하도록). 본 문서 §8 sim이 새 D_total 분포를 입력받아 holo_factor를 재산출하는 경로 제공.
- **주의(설계 사실)**: D항은 reachable(현실)에선 훨씬 작다 — D항 메커니즘은 L10(정보층, dec25+)에서만 활성(systems §2-I)이라 dec25 이전 D항=0, 곡선 B codex도 램프업. reachable 단축 = −18.7%(economy §7.2.3, worst −28.5% 대비 큰 여유). **그러므로 worst-case(상수 PT1)는 보수적 상한이고, 실제 출시 리스크는 밸런스 미세조정 사안**(scope §5-1 판정 B). 그래도 magnitude 실측 없이는 가정이므로 수집 필수.

### 4.4 캠페인 런 시간 분포 — 단조성 실증

- **측정**: `pt_run_time` = 각 상전이 PT_k의 신영역 능동 엔진-시간, **run_index별로 분리**.
- **목표 분포** (economy §1.2 / §4.4, 본 문서 §8 monotone=True 확인):
  - run1 (첫 캠페인): PT1~PT6 = [4.26, 7.18, 9.60, 15.36, 20.60, 23.60]h — **단조 증가**.
  - run2 (재하강): [1.55, 3.66, 6.56, 12.10, 18.39, 22.65]h — **단조 증가**(누적 QF로 전체 빨라지나 런 내 단조 보존).
- **재튜닝 트리거 (race 붕괴 감지)**: **어떤 PT_k 런타임이 직전 PT_{k-1}보다 짧으면 = 부스트가 인플레를 추월(race 붕괴, PR4)**. economy §1.3이 base_rate=0.25에서 단조 보장하나, 실제 플레이어 구매 휴리스틱이 다르면(고티어 우선 가정과 다르면, economy §5.2) ±20% 흔들릴 수 있음 → 비단조 발생 시 base_rate 또는 벽 스케줄 재검토.
- **per-run_index 추세**: 재하강 회차가 올라갈수록 전체 런타임이 짧아지는 곡선(누적 QF 가속)이 실측되는지. economy §4.4 예측(run1→run2 단축)의 실증.

## 7. 개인정보 / 프라이버시 (수집 최소화·익명화)

> 법문(개인정보처리방침)은 business-legal 체크리스트(task #21) 담당. 본 절은 **수집 설계의 프라이버시 원칙**(무엇을 수집/비수집, 동의 모델).

### 7.1 핵심 원칙
- **opt-in 동의 (기본 OFF)**: 첫 실행 시 명시적 동의 요청. **동의 전 0 전송.** 거부해도 게임 100% 플레이 가능(텔레메트리는 게임 기능 아님). GDPR/한국 개인정보보호법 정합 — opt-out이 아니라 opt-in.
- **수집 최소화**: 페이싱·리텐션 검증에 *필요한 최소*만. "혹시 몰라" 수집 금지. 본 문서 §2~5에 열거된 지표 외 수집 금지.
- **익명화**: 
  - **개인식별정보(PII) 0 수집** — 이름·이메일·IP(저장 안 함)·정확 위치 없음.
  - 식별자 = **익명 설치 UUID**(랜덤 생성, 기기-게임 1쌍, 개인 역추적 불가). Steam ID·계정과 **연결 금지**.
  - 리텐션 코호트는 UUID 기준(개인 아님). 세그먼트는 인게이지먼트 스타일 등 행동 군집만.
- **세이브 내용 비전송**: 세이브 데이터 자체는 절대 전송 안 함(§5.4). 손상 *이벤트 카운트*·집계 *수치*만.

### 7.2 수집 데이터 분류

| 분류 | 수집 | 예 |
|---|---|---|
| **수집 (익명 집계)** | O | 이벤트 타임스탬프(상대), PT/크런치/재하강 도달, D_total·완성도 *수치*, 세션 길이·빈도, fps·메모리 추세, 손상 이벤트 카운트 |
| **비수집 (PII)** | X | 이름, 이메일, IP 저장, 정확 위치, Steam/계정 ID 연결, 세이브 원문, 디바이스 고유 식별자(MAC 등) |

### 7.3 전송·보관 (경량·프라이버시 우선)
- **로컬 버퍼 후 배치 전송**: 매 이벤트 즉시 전송 금지(배터리·네트워크·프라이버시). 로컬 큐에 모아 세션 종료/주기적 배치. 오프라인이면 다음 접속 시.
- **전송 최소 페이로드**: 익명 UUID + 이벤트 코드 + 수치. 자유 텍스트·식별 가능 메타 금지.
- **옵트아웃·삭제 경로**: 설정에서 언제든 OFF + "내 데이터 삭제 요청"(UUID 기준 서버 삭제). tech-arch §1.1 settings 네임스페이스에 동의 상태 저장.
- **보관 기간 한정**: 집계 후 원시 이벤트는 기간 한정 보관 후 폐기(법무 정책 연계).

### 7.4 구현 경량성 (수집 코드 v1.0 IN — scope C4)
- **이벤트 버스 구독으로 비침습 수집**(tech-arch §4.3 event-hub): 게임 로직에 텔레메트리 코드 직접 삽입 금지. 도메인 이벤트(상전이·크런치·발견)를 구독해 텔레메트리 모듈이 자율 기록 → 코어 오염 0, 제거/비활성 용이.
- **stats 네임스페이스 활용**(tech-arch §1.1): D_total·완성도·카운터는 이미 stats에 추적되는 값 → 텔레메트리는 그 스냅샷만 읽음(추가 계산 0).
- **no-op 폴백**: 동의 OFF이거나 전송 실패 시 완전 no-op(게임 영향 0). 웹/Steam 빌드 공통.

## 8. 가정 · 한계 · 재현 sim

### 8.1 재현 (목표 분포 도출 근거)
본 문서의 목표 수치는 economy.md 시뮬 출력 + 아래 재현 계산에 근거:
- **PT1 시간·크런치 캘린더**: economy.md §1.2·§4.2·§4.3 (campaign6.py / final_verify.py / bigcrunch.py 출력).
- **D_total 안전 밴드 표**(§4.3): 본 분석 패스 재현 — D항=log10(D+1)×0.008, worst-case H=(1+D항)×1.35, 단축=1−1/H. 1e7=−29.9%(경계), 1e8=−30.4%(위반). economy §7.2.3 표와 정합.
- **능동/방치 진행초 비율**(§4.2): 기본 플레이어 능동 0.5h/일 vs 오프라인 크레딧 15.6h/일 = 97% 방치 → "60/40은 진행분할 아님" 결론.
- **런타임 단조성**(§4.4): run1/run2 시퀀스 monotone=True 확인(economy §1.2/§4.4 값).

### 8.2 가정 (미검증)
- **리텐션 목표(D1≥35%/D7≥15%/D30≥6%)**: 방치형 업계 벤치 + economy 페이싱 도출. **실측 전 전부 가정.** 출시 후 첫 코호트로 교체 필수.
- **체크인 모델(2회/일, 15분)**: economy 전제. 실제 세션 간격 분포로 교체(§2.3).
- **구매 휴리스틱(고티어 우선)**: economy §5.2 가정. 실제 전략이 다르면 런타임 단조성·PT1 분포 ±20% 흔들림 → §4.4가 이를 실측.
- **D_total magnitude**: D 생산률 미확정(research-tree TBD)이라 **현재 완전 미검증** — 이 수집이 그것을 닫는 목적(§1-A).

### 8.3 한계 / 후속
- **목표 분포는 단일 플레이어 모델 기반**. 실제는 플레이어 다양성(스타일·디바이스·세션)으로 분산이 큼 → 분위수(p50/p90/p99)로 봐야 하며, 평균은 오해 소지.
- **곡선 B reachable 단축**(§5.2)은 완성도-진행 상관 미확정으로 front-loaded worst(−17.7%)로 보수 산정. 도감 완성도 분포 실측 후 reachable 재계산.
- **분석·대시보드 구체 구현**(전송 백엔드·집계 파이프라인·시각화)은 출시 후 + game-programmer/운영. 본 문서는 지표 정의·목표·임계까지.
- **A/B 테스트 프레임**: EA 기간 파라미터 A/B(예: 첫 벽 dec19 vs dec20)는 향후 확장. v1.0은 단일 빌드 수집부터.
