## 0. 이 문서의 위치 (중복이 아니라 구체화)

세 문서가 같은 4정거장을 다른 해상도로 본다. **겹치지 않는다 — 줌 레벨이 다르다.**

| 문서 | 질문 | 해상도 |
|---|---|---|
| `roadmap.md` | **무엇을, 어떤 순서로** | 정거장·마일스톤(M1.1~M3.9), 상대 규모(작음/중간/큼), 졸업 기준 |
| `reality-check.md` | **정말 할 수 있나, 얼마나** | 정거장별 공수 *범위*, 헌신 크기, 번아웃 지점, 솔로 분담 한계 |
| **`dev-schedule.md`** (이 문서) | **어떤 작업 단위로, 무엇이 무엇을 막나, 다 됐다는 증거는** | 작업 단위(W) · 의존성 그래프 · 임계 경로 · 병렬 레인 · 작업별 DoD · 추적 방법 |

> **읽는 순서:** roadmap에서 "지금 어디고 다음이 뭔지"를 보고 → reality-check에서 "이게 얼마나 큰 헌신인지"를 각오하고 → **이 문서에서 "이번 주에 손댈 작업 한 칸과 그게 끝났다는 정의"를 집는다.** 이 문서는 *실행 트래커*다.

### 0-1. ⚠️ 절대 일정을 단정하지 않는 이유 (reality-check §0 계승)

이 문서는 **"몇 월에 끝난다"를 적지 않는다.** reality-check가 못박았듯, 처음 만드는 사람의 95%가 정거장 1을 절반 이하로 추정한다(§1-A). 절대 날짜는 환상을 심고, 환상은 6개월 뒤 더 크게 무너진다(PR2 번아웃의 연료).

대신 이 문서는 **순서·의존성·완료 정의**만 확정한다. 그것들은 시간이 가도 안 변한다. "언제"는 당신이 주당 몇 시간을 넣느냐의 함수이지(reality-check §1-E: 달력시간 = 실투입 × 40/주당시간), 이 문서가 정할 수 있는 게 아니다.

- **상대 규모(roadmap)는 유지**: 각 작업에 `XS/S/M/L` 태그. 절대 시간이 아니라 *서로 간의 크기 비*.
- **범위는 reality-check 참조**: 정거장 1 = 3~5개월치 실투입(현실), 정거장 1→3 = 12~24개월치(보통). 이 문서는 그 범위를 *작업으로 쪼개기*만 한다.

## 1. 작업 단위 명명 규칙 (어떻게 추적하나)

roadmap의 마일스톤(M1.1 등)을 **추적 가능한 작업 단위(W = Work item)**로 분해한다.

- **`W{정거장}.{마일스톤}.{작업}`** 형식. 예: `W1.1.a` = 정거장1 / M1.1(F-게이트) / 첫 작업(세이브 봉투).
- 각 작업 단위는 **하나의 완료 정의(DoD)**를 가진다. DoD = "이게 끝났다고 말할 수 있는 검증 가능한 조건". 체크박스로 추적.
- **규모 태그**: `XS`(반나절급) `S`(며칠급) `M`(1~2주급) `L`(여러 주급). *상대값* — reality-check가 경고한 과소평가를 막기 위해 한 칸을 의심스러우면 한 단계 키워 잡아라.
- **레인 태그**: `[코어]`(직렬 의존, 막히면 정거장 정체) / `[병렬]`(코어 막힐 때 전환 가능) / `[외부]`(외주/에셋 — reality-check §3-B).

> **솔로 과부하 방지:** 작업 단위는 의도적으로 작게 쪼갰다. risk PR2 완화책("작은 마일스톤으로 완료 도파민을 자주")의 실천이다. 한 작업이 `L`이면 → 다시 반으로 쪼갤 수 있는지 먼저 본다. "M1.1을 다시 반으로"(reality-check §2-B ①)가 번아웃 한가운데의 생존법.

### 3-B. 정거장 1 졸업 게이트 (roadmap §1-C / scope §2 그대로)

이 5개가 충족돼야 정거장 2로. **통과 못 하면 콘텐츠를 더 만들 때가 아니라 코어 루프를 고칠 때다(분기점).**

- [ ] G1.1 — 첫 30초에 "작아짐=강해짐"이 화면에서 느껴진다.
- [ ] G1.2 — 자동화 전 1~2분 능동 클릭이 지루하지 않다.
- [ ] G1.3 — 첫 상전이가 "해냈다 + 또 하고 싶다"를 만든다. **(약하면 멈춤)**
- [ ] G1.4 — 닫고 켜니 오프라인 진행이 기분 좋다.
- [ ] **G1.5 — 개발자 본인이 "한 판 더" 하고 싶다. (가장 중요)** **(약하면 멈춤)**

> **검증 방법:** G1.3·G1.5는 주관적이라 자기기만 위험 → **qa-balancer 플레이테스트 시뮬로 객관 확인 권장**(roadmap §1-C). reality-check §4-B: 이 졸업은 *통과 의식이 아니라 계속/중단을 정직하게 묻는 분기점*. **들인 게 가장 적을 때 분기하는 게 가장 싸다.**

| **W3.8.b 텔레메트리 수집** | 출시 후 holo_factor 재튜닝 | 수집 없으면 영영 데이터 없음(R8) |
| 모든 정거장 3 콘텐츠·인프라 | M3.9 최종 QA | 마지막 직렬 마감 |

### 7-3. ★ 임계 경로 (Critical Path — 전체를 가장 길게 결정하는 직렬 사슬)

> 임계 경로 = 이 사슬의 어느 칸이 늦으면 *전체 출시가 그만큼 늦는다*. 병렬 작업으로 우회 불가능한 뼈대.

```
F-게이트(M1.1: 특히 W1.1.c 루프 = 가장 가파른 학습)
  → M1.2 코어 엔진 (정거장 1 일의 70%, reality-check "일의 대부분")
  → M1.5 첫 상전이 (클라이맥스, 상태전이 벽)
  → 【정거장 1 졸업 G1.3·G1.5】 ★ 최대 분기점
  → 정거장 2 W2.4 FTUE (모든 폴리시 통합)
  → 【정거장 2 졸업: GIF】
  → M3.2 빅 크런치 → M3.3 재하강 (콘텐츠 코어 직렬)
  → M3.9 최종 QA (마지막 게이트)
  → 【정거장 3 졸업 = 출시】
```

**임계 경로의 의미:**
- **F-게이트 + M1.2가 임계 경로의 가장 비싼 구간**(reality-check §1-A: 가장 가파른 학습 곡선). 여기를 빨리·정확히 넘는 것이 전체 일정의 최대 레버.
- **도감·연구·아트·사운드·Steam 패키징은 임계 경로가 *아니다*** → 코어가 막힐 때 이쪽으로 전환해도 전체 일정이 안 밀린다(오히려 대기시간 활용). 이것이 §8 병렬 레인의 일정상 가치.
- **3개 졸업 게이트가 임계 경로의 관문.** 특히 정거장 1 졸업(G1.3·G1.5)은 *일정 관문이자 계속/중단 분기*(reality-check §4-B).

## 8. ★ 병렬 가능 작업 (코어 막힐 때 전환 / 팀이면 분담 라인)

> reality-check §3·roadmap §6 핵심: **막히면 레인을 바꿔라.** 솔로라도 코어가 안 풀릴 때 *독립 작업*으로 전환하면 진행감을 잃지 않는다(번아웃 방어). 팀이면 동시 분담.

### 8-1. 정거장별 병렬 레인

| 정거장 | `[코어]` 직렬 (막히는 곳) | `[병렬]` 전환 가능 (코어 독립) | `[외부/외주]` (다른 손) |
|---|---|---|---|
| **1** | F-게이트 → M1.2 → M1.5 → M1.6 | **W1.7.b 도감 데이터 / W1.7.c 연구 노드** (데이터 주입, 코어 불간섭) · W1.4 손맛 폴리시(엔진 후 일부) | — (정거장 1은 못생겨도 됨) |
| **2** | W2.4 FTUE 통합 | W2.6 주스 · W2.5 보이스(content) | **W2.1 아트 · W2.2 사운드** (reality-check §3-B 외주 1순위) |
| **3** | M3.1→M3.2→M3.3 (콘텐츠 코어) · M3.9 QA | **M3.4 도감87 / M3.5 연구트리** (데이터) · **M3.7 Steam패키징 / M3.8 인프라** (platform 격리) | **M3.6 아트·사운드 6층** (외주 확장) · 영어 번역 |

### 8-2. 솔로 레인 전환 규칙 (reality-check §2-C / risk PR2)

1. **코어가 막히면 → 데이터 레인으로.** 정거장 1: 도감 입자 데이터(W1.7.b)·연구 노드(W1.7.c)를 `data/`에 채운다. "표 채우기에 가깝다"(reality-check §3-A, 가장 할 만한 영역).
2. **데이터도 막히면(또는 다 했으면) → 플레이버/폴리시로.** 손맛 피드백 미세조정, UI 정리.
3. **그것도 막히면 → 휴식.** 번아웃이 1인 개발 최대 적(risk PR2). 멈춤도 전략.

> **데이터 레인이 왜 안전한가:** 수치·설계가 이미 닫혀 있다(economy/content 확정, scope §0). 도감·연구는 *설계*가 아니라 *주입*이다 → 코어 디버깅으로 지친 머리로도 진행 가능. 이것이 "막힐 때 갈 곳"이 항상 있다는 보장.

### 8-3. 팀(2인+) 분담 라인

솔로가 아니면 §8-1의 `[병렬]`·`[외부]` 열이 그대로 동시 분담 라인:
- **A(코어 프로그래머):** 임계 경로(F-게이트→M1.2→상전이→재하강) 전담.
- **B(콘텐츠/데이터):** 도감·연구 데이터(W1.7.b·c, M3.4·M3.5) + 플레이버.
- **외주:** 아트(캡슐·VFX)·사운드·영어 번역(reality-check §3-B). 정거장 2 진입 전 계약.

## 10. 추적 방법 (솔로 기준 과하지 않게)

> 솔로에게 무거운 이슈 트래커는 또 다른 번아웃 원천. **경량·체크리스트 우선.**

### 10-1. 권장: 이 문서 = 트래커 (가장 가벼움)

- **이 문서의 작업 단위 표가 곧 트래커.** 각 작업에 상태 마커를 직접 단다: `[ ]` 미착수 / `[~]` 진행 / `[x]` 완료 (asset-list.md와 동일 규약).
- **졸업 게이트 체크박스(§3-B·§4-C·§5-B)가 정거장 진척의 단일 신호.** 5개/4개/6개 체크되면 다음 정거장.
- **DoD가 "완료" 판정.** 주관적 "다 한 것 같다"가 아니라 DoD 조건 충족으로 판정 → 자기기만·미완성 출시 방어.

### 10-2. 선택: 경량 칸반 (팀이거나 작업이 많아지면)

- GitHub Projects / 단순 칸반(To Do / Doing / Done) 3열. 작업 단위(W)를 카드로.
- **WIP 제한 1~2**: 솔로는 동시에 1개 코어 작업 + 1개 병렬 레인까지만. 그 이상 벌이면 스코프 크립·번아웃 신호.

### 10-3. 주간 셀프 체크 (5분, 번아웃 조기경보)

매주 스스로에게:
- [ ] 이번 주 작업 단위 1개라도 `[x]` 됐나? (안 됐으면 → 막힌 곳 확인, 레인 전환?)
- [ ] 한 작업에 비정상적으로 오래 박혀 있나? (→ 더 잘게 쪼개거나 에이전트 위임)
- [ ] 백로그가 늘기만 하고 v1.0 IN으로 안 빠지나? (→ PR1 스코프 크립 적신호)
- [ ] 정거장 1 졸업까지 거리감이 멀어지고 있나? (→ 콘텐츠 줄이고 코어 집중)

> **추적의 목적은 통제가 아니라 *조기경보*다**(risk §0: "신호를 일찍 잡으면 싸게 막는다"). 진척이 멈춘 신호를 일찍 보면 싸게 고친다.
