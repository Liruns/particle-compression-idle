## ★ 0. 계정 필요성 결정 (가장 먼저)

### 결정: **v1.0은 커스텀 계정/로그인 서버를 만들지 않는다.**

신원은 **Steam(데스크톱)**, 저장은 **로컬 + Steam Cloud**, 휴대는 **수동 export/import 문자열**로 해결한다. 자체 계정 DB·인증·백엔드 서버 = **전부 없음.**

| 신원·저장 책임 | v1.0 방식 | 근거 |
|---|---|---|
| **데스크톱 신원** | **Steam 계정**(이미 로그인된 것) | Steam에서 사는 순간 신원은 Valve가 보증. 우리가 또 만들 이유 없음 |
| **데스크톱 저장·동기화** | 로컬 `fs` + **Steam Cloud**(자동) | 계정당 클라우드 무료 제공(§2). 크로스-PC 싱크가 *공짜로* 따라옴 |
| **웹 신원** | **없음(익명, 기기 로컬)** | itch.io/브라우저는 로그인 없이 즉시 플레이. 세이브는 그 브라우저 localStorage에 귀속 |
| **웹↔데스크톱 휴대** | **수동 export/import**(Base64 문자열) | 계정 없이도 기기 이전 가능(§2.2). 봉투 포맷 그대로 복붙 |

### 왜 계정을 안 만드는가 (근거)

1. **장르가 오프라인 싱글이다.** GDD §2: "순수 방치 / 오프라인 1급 시민". 멀티플레이·실시간 경쟁·서버 권위 진행이 **하나도 없다.** 계정 서버가 보호할 자산(거래·랭킹 신뢰)이 없음.
2. **진행은 클라이언트가 진실이다.** 오프라인 계산·세이브가 전부 로컬에서 결정(tech-arch §1·§3). 서버 검증이 의미 없는 구조 — 계정을 둬도 "서버가 진행을 지킨다"가 성립 안 함.
3. **계정 서버는 순비용·순리스크다.** 인증·DB·가용성·보안·개인정보 처리의무가 전부 *추가*되는데, 게임이 주는 가치는 0. `business-legal-checklist.md` §5(개인정보처리방침·GDPR·COPPA)의 부담만 키운다.
4. **개인정보 최소수집 원칙과 정합.** `analytics-plan.md`는 이미 **익명 설치 UUID만, Steam ID·계정과 연결 금지, PII(이메일·계정 연결) 비수집**을 못박았다. 자체 계정 = 이메일·로그인 = 이 원칙을 정면으로 위반. **계정 없음이 프라이버시 설계와 한 방향.**
5. **Steam이 신원·클라우드를 무료로 준다.** 데스크톱에서 필요한 "내 진행을 내 다른 PC에서"는 Steam Cloud로 끝난다(§2). 바퀴를 다시 발명할 이유 없음.

### 결정의 한 줄
> **커스텀 계정 없음 = Steam 신원 + 로컬/Cloud 저장 + 수동 export.** 오프라인 싱글 장르에 계정 서버는 보호할 것 없는 순비용이며, 익명 텔레메트리 원칙(`analytics-plan.md`)과도 한 방향이다.

## 1. 신원·저장 모델 (빌드별)

세 가지 "내가 누구이고 내 세이브가 어디 있나"가 빌드마다 다르다. **추상화는 StorageAdapter(tech-arch §1.6)가 흡수** — 게임 로직은 차이를 모른다.

| 빌드 | 신원 | 세이브 위치 | 자동 동기화 | 휴대 수단 |
|---|---|---|---|---|
| **웹(itch.io/브라우저)** | 익명(기기·브라우저 귀속) | localStorage (+IndexedDB fallback) | 없음 | export/import 문자열 |
| **Steam(데스크톱)** | Steam 계정 | `fs` 파일(Steam Cloud 경로) | **Steam Cloud**(자동, 크로스-PC) | export/import + Cloud |

- **웹 세이브는 휘발 위험 인지**: 브라우저 캐시 삭제·시크릿 모드·기기 교체 시 사라질 수 있다 → **export 유도 UX 필수**(§3-B). 웹은 "플레이테스트/맛보기" 채널이고, 영속 진행의 본진은 Steam(GDD §2).
- **Steam 세이브가 진행의 정본(canonical)**: 구매 후에는 Cloud가 자동 백업·크로스-PC를 책임.

## 2. 동기화 정책

### 2.1 Steam Cloud (데스크톱 크로스-PC) — *상세 메커니즘 tech-arch §1.5*

- `fs` 세이브 경로를 Steam Cloud에 매핑 → **PC A에서 플레이 → PC B에서 이어하기**가 자동(계정당 무료 용량, 우리 세이브는 압축 후 수십~수백 KB라 여유).
- **충돌 해소(같은 계정, 두 PC에서 따로 진행)**: 세이브 `lastSave` timestamp **최댓값**을 정본으로 채택(tech-arch §1.7). "더 최근에 저장된 것이 이긴다" — 단순·예측가능. 충돌 발생은 `analytics-plan.md`가 `클라우드 충돌 롤백`으로 계측.
- **변조 방어와 동일 소스**: `lastSave`는 시계조작 방어(tech-arch §1.7·§3.2)와 충돌 해소에 **같은 값**을 쓴다 → 메커니즘 일원화.

### 2.2 웹 ↔ Steam 이전 — *상세 메커니즘 tech-arch §1.8*

- **수단: 수동 export/import**(봉투를 Base64 문자열로 클립보드 복붙). tech-arch §1.8이 정한 그대로.
- **왜 자동이 아닌가**: 웹(localStorage)과 NW.js(파일)는 **물리적으로 분리된 저장소**이고, 둘을 잇는 계정/서버가 없으므로(§0) 자동 브리지가 불가. 수동 문자열이 계정 없이 기기를 잇는 유일·충분한 방법.
- **방향**: 주로 웹(맛보기) → Steam(정본)으로의 *1회성 이주*를 상정. 역방향(Steam→웹)도 같은 문자열로 가능하나 비주류.
- **호환성**: export 문자열은 세이브 봉투(`{version,data,checksum}`) 그대로라, 받는 쪽이 구버전이면 **마이그레이션 체인(tech-arch §1.3)이 자동 흡수**. 버전 충돌로 깨지지 않음.

### 2.3 충돌 해소 규칙 한 줄
> **`lastSave` 최댓값 = 정본**(tech-arch §1.7). Cloud 충돌도, 시계조작 방어도, export 병합도 전부 이 단일 규칙 — 흩어진 정책 없음.

## 3. 진행도 복구 — 사용자 관점 플로우

기술 메커니즘(롤링 백업·체크섬·원자적 쓰기)은 tech-arch §1.8이 보유. 여기서는 **사용자가 겪는 상황별 복구 단계**를 정의한다(ux가 화면으로 구현).

### 3-A. 세이브 손상 (체크섬 불일치 / 반쪽 쓰기 / Cloud 충돌)
사용자가 보는 흐름:
1. 게임 시작 → 메인 세이브 로드 시 **체크섬 검증 실패** 감지.
2. 게임이 **자동으로 롤링 백업(직전 2~3세대)** 중 가장 최근의 유효본으로 복구 시도(tech-arch §1.8). *사용자 개입 없이* 대부분 여기서 해결.
3. 백업도 전부 실패 시 → 손상 원본을 `*.corrupt.bak`로 **보존**(절대 침묵 삭제 안 함, tech-arch §1.3)하고, 사용자에게 **"세이브 복구 실패. 백업 파일을 보존했습니다 + export 문자열이 있으면 붙여넣어 복구하세요"** 안내.
4. 계측: `save_corrupt`·`migration_fail`·`.corrupt.bak 생성`을 `analytics-plan.md`가 익명 집계 → 손상 빈도 모니터링.

### 3-B. 세이브 분실 (기기 교체 / 브라우저 캐시 삭제 / 재설치)
- **Steam(데스크톱)**: Steam Cloud가 자동 복원 → 같은 계정으로 재설치/다른 PC 로그인하면 **그냥 이어진다.** 사용자 액션 0.
- **웹**: 자동 복원 수단 없음(계정 없으므로). → **export 문자열로만 복구.** 그래서:
  - **사전 예방 UX(핵심)**: 의미 있는 진행(예: 첫 상전이) 도달 시 **"진행을 백업하세요"** export 유도. 웹 사용자에게 export는 *선택이 아니라 권장 습관*으로 노출(ux).
  - 분실 후: export 문자열 보유 시 → 설정의 **import**에 붙여넣기 → 복구 완료.

### 3-C. 기기 이전 (정상적 이사)
| 시나리오 | 단계 |
|---|---|
| **PC → PC (둘 다 Steam)** | 자동. 새 PC에서 Steam 로그인 → Cloud가 세이브 내림 → 이어하기. |
| **웹 → Steam** | 웹에서 **export** → Steam 빌드 설정에서 **import** → 정본이 Steam(Cloud)으로 승격. |
| **웹 브라우저 → 다른 브라우저/기기** | 기존 브라우저에서 **export** → 새 브라우저 **import**. (Cloud 없음) |
| **Steam → 웹** | Steam에서 **export** → 웹 **import**. (비주류, 가능) |

### 3-D. 복구 안전망 요약 (어떤 보호가 언제 작동하나)
| 위협 | 1차 방어(자동) | 2차 방어(사용자) | 메커니즘 위치 |
|---|---|---|---|
| 파일 손상 | 롤링 백업 자동 복구 | export import | tech-arch §1.8 |
| 반쪽 쓰기 | 원자적 쓰기(임시→교체) | — | tech-arch §1.8 |
| 기기 분실(Steam) | Steam Cloud 자동 | — | §2.1 |
| 기기 분실(웹) | — | export import(사전 백업 전제) | §2.2 |
| Cloud 충돌 | `lastSave` 최댓값 채택 | — | §2.1 |
| 시계조작 | 48h 클램프 + lastSave 최댓값 | — | tech-arch §1.7·§3.2 |

> **원칙(tech-arch §1과 공유): 세이브는 신성하다.** 어떤 복구 경로도 **데이터를 침묵 삭제하지 않는다** — 최악의 경우에도 `*.corrupt.bak` 보존 + 사용자 안내.

## 4. 미래 여지 (v1.0 구조가 막지 않게)

지금 안 만들지만, 나중에 필요해지면 **현재 결정이 길을 막지 않도록** 경로만 확보한다.

| 미래 기능 | 언제 필요할 수 있나 | v1.0이 막지 않는 이유 + 마이그레이션 경로 |
|---|---|---|
| **리더보드 / 랭킹** | 커뮤니티가 "가장 빠른 빅 크런치" 경쟁을 원할 때 | **Steam Leaderboards(steamworks.js)로 충분** — 커스텀 계정 불필요. 진행 통계는 이미 `stats` 네임스페이스(tech-arch §1.1)에 있음. 이벤트 버스(tech-arch §4.3) 구독으로 점수 제출만 추가. *계정 서버 여전히 0.* |
| **크로스플랫폼 클라우드(웹↔모바일↔Steam 자동 싱크)** | 웹/모바일을 정식 영속 플랫폼으로 승격할 때 | **그때 비로소 계정/백엔드 도입을 고려.** 단 세이브가 이미 **봉투(`{version,data,checksum}`) + StorageAdapter 추상화**(tech-arch §1.2·§1.6)라, "CloudSyncAdapter"를 어댑터 한 종 추가하는 형태로 얹힘 — 게임 로직 무수정. export 포맷이 그대로 전송 단위가 됨. |
| **계정 연동(선택적 로그인)** | 위 둘이 동시에 강하게 요구될 때 | StorageAdapter/Steam 인터페이스가 이미 `platform/`에 격리(tech-arch §5.2)되어, 신원 제공자를 *주입 교체*하는 자리가 비어 있음. 코어는 "누가 로그인했나"를 모르므로 추가가 국소적. |
| **세이브 서버 백업(분실 대비 자동 클라우드)** | 웹 사용자 세이브 휘발이 리텐션 문제로 드러날 때 | 동일하게 어댑터 추가. **단 도입 시 개인정보 처리의무 발생**(`business-legal-checklist.md` §5) → 비용·법무 재검토 필수. 그래서 *데이터가 필요를 증명하기 전엔 안 함.* |

### 미래 여지 원칙 한 줄
> **v1.0의 "계정 없음"은 막다른 길이 아니라 *미루기*다.** 봉투 포맷 + StorageAdapter + platform 격리 = 나중에 계정/클라우드를 **어댑터 추가로** 얹을 수 있는 구조. 단, 도입은 *데이터가 필요를 증명할 때*만(특히 개인정보 부담을 지는 서버 백업).

## 5. 요약 (핵심 4줄)

1. **★ 계정 결정: v1.0은 커스텀 계정/로그인 서버 없음.** 신원=Steam(데스크톱)/익명(웹), 저장=로컬+Steam Cloud, 휴대=수동 export/import. 오프라인 싱글 장르라 서버가 보호할 자산이 없고, 익명 텔레메트리 원칙(`analytics-plan.md`)과도 한 방향.
2. **동기화: Steam Cloud가 데스크톱 크로스-PC를 자동 처리**, 웹↔Steam은 export/import 문자열, 충돌은 `lastSave` 최댓값(tech-arch §1.7) 단일 규칙으로 해소.
3. **복구: Steam은 Cloud 자동 복원 / 웹은 export 사전백업이 생명줄.** 손상 시 롤링 백업 자동 복구→실패해도 `*.corrupt.bak` 보존(침묵 삭제 절대 금지). 사용자 관점 플로우는 §3.
4. **미래 여지: 리더보드는 Steam으로 충분(계정 0), 크로스플랫폼 클라우드는 봉투+StorageAdapter에 어댑터 추가로 얹힘** — v1.0 구조가 막지 않되, 서버 도입은 데이터가 필요를 증명할 때만.
