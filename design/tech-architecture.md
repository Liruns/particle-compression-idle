## 1. 세이브 시스템

세이브는 신성하다. 며칠~수 주 누적된 진행이 단 한 번의 손상·마이그레이션 실패로 증발하면 방치형 장르에서 치명적이다. 세 가지 위협을 동시에 막는다: **(A) 포맷 진화로 인한 구버전 깨짐, (B) 파일/문자열 손상, (C) 변조(시계조작·세이브 편집).**

### 1.1 상태 스키마 구조 — 무엇을 저장하는가

저장 대상은 "게임의 진실 상태(source of truth)" 전부이되, **파생 캐시는 저장하지 않는다**(로드 후 재계산). 스키마는 도메인별 네임스페이스로 분할해 마이그레이션 단위를 작게 유지한다.

| 네임스페이스 | 포함 상태(개념) | 비고 |
|---|---|---|
| `meta` | 스키마 `version`, 게임 빌드 버전, 생성/최종저장 timestamp, 총 플레이타임 | 마이그레이션·변조 탐지의 기준 |
| `resources` | `E`, `C`, `D_current`, `QF`, `lifetime_C`, `D_lifetime` | **모두 Decimal 직렬화**(§2.4). `lifetime_C`·`D_lifetime`은 재하강 보존 키 |
| `chain` | 8단 압축기 각 티어의 보유 개수(`bought`) | 비용·생산량은 파생 → 저장 안 함, 로드 시 재계산 |
| `prestige` | 상전이 횟수(`N_prestige`), 빅 크런치 회차(`run_index`), `QF_claimed`, 현재 집중 서브층 선택 | 재하강 차별화(systems §2-K) 상태 |
| `layers` | 현재 층/서브층 인덱스, 층별 메커니즘 상태(위상 state, 스핀 네트워크 그래프, 핵력 게이지 등) | systems §2 메커니즘별 직렬화 필요 항목. **리셋 매트릭스(§5)에서 리셋 대상이라도 "현재 런 진행 중"이면 저장** |
| `codex` | 발견된 입자 ID 집합(87개 중), 발견 timestamp | 영구 보존. 완성도는 파생(개수/76) |
| `research` | 구매한 연구 노드 ID 집합(52개 중) | 영구 보존 |
| `settings` | 오프라인 정밀모드 on/off, 숫자 표기법, 사운드/그래픽 옵션 등 | 게임 진행과 분리, 마이그레이션 영향 최소 |
| `stats` | 누적 통계(총 압축 횟수, 총 상전이, 도전과제 카운터 등) | **정확 카운팅 항목은 일반 정수**(Decimal 아님, research gotcha) |

**원칙:**
- **진실/파생 분리.** 비용 곡선 `cost_k(n)`, 생산량 `dg_k/dt`, `production_mult`, `dec`, `r`, `holographic_mult` 등은 전부 파생값 → **저장 금지, 로드 후 재계산**. 저장 용량·마이그레이션 표면적·불일치 위험을 동시에 줄인다.
- **집합은 ID 배열로.** 도감/연구는 "어떤 것을 가졌나"의 집합이므로 boolean 87개/52개 배열이 아니라 **발견된 ID 리스트**로 저장(스파스, 새 엔트리 추가에 강건).
- **메커니즘 상태의 직렬화 책임은 각 메커니즘 모듈이 진다.** 스핀 네트워크 그래프(노드·엣지)처럼 복잡한 상태는 모듈이 `serialize()/deserialize()` 인터페이스를 노출하고, 세이브 시스템은 그 결과(평범한 JSON 값)만 봉투에 담는다. 세이브 시스템이 메커니즘 내부를 알 필요 없음.

### 1.2 세이브 봉투(envelope) 포맷

최상위는 **항상** 다음 3-필드 봉투다. 이 구조는 프로토타입 1일차부터 고정한다(나중에 추가하면 구버전 마이그레이션이 불가능해짐 — research gotcha).

```
{
  version:  <정수>        // 스키마 버전. 마이그레이션 디스패치 키
  data:     <문자열>      // 위 스키마를 직렬화+압축한 페이로드
  checksum: <문자열>      // data에 대한 무결성/변조 탐지 값
}
```

- **`version`은 봉투 바깥에 평문으로 둔다.** 압축/암호화 안에 넣으면 "어느 버전인지 알려면 먼저 풀어야 하는데, 어떻게 풀지는 버전이 정한다"는 순환이 생긴다. 버전은 항상 즉시 읽혀야 한다.
- **`data`는 평문 JSON을 lz-string으로 압축한 것**(§1.4 인코딩 통일). 암호화하지 않는다 — 접근성 우선 장르이며(research gotcha "치팅 방지 보통 생략"), 변조 탐지는 체크섬으로 충분.
- **`checksum`은 변조 탐지용이지 보안용이 아니다.** 클라이언트 사이드 게임에서 진짜 변조 방지는 불가능. 목표는 "악의적 크래커 차단"이 아니라 **우연한 손상 검출 + 캐주얼 편집 억제 + 클라우드 충돌 시 유효성 판정**. (구현 디테일은 game-programmer; 체크섬 알고리즘 선택은 "빠르고 결정적"이면 됨.)

### 1.3 버전 관리·마이그레이션 전략 — 세이브 깨짐 방지

**핵심 규칙: 마이그레이션은 단방향 체인이다.** `migrate_v1_to_v2`, `migrate_v2_to_v3` … 식의 작은 변환 함수를 순서대로 등록하고, 로드 시 `save.version`부터 현재 버전까지 **순차 적용**한다. "v1에서 v5로 한 번에" 같은 점프 변환을 만들지 않는다(조합 폭발 방지, AD `convertVersion` / Evolve 선례).

```
로드 흐름(개념):
  envelope = parse(raw)
  if envelope.version > CURRENT_VERSION: → "더 새로운 세이브" 경고, 안전 거부(다운그레이드 방지)
  decompressed = decompress(envelope.data)
  state = JSON.parse(decompressed)
  while state.version < CURRENT_VERSION:
      state = migrations[state.version](state)   // v→v+1 한 칸씩
  validate(state)                                // 누락 필드 기본값 채움, NaN/null 방어
```

**규칙:**
- **마이그레이션 함수는 한 번 작성하면 영원히 보존**(과거 세이브를 가진 유저는 언제든 돌아올 수 있다). 삭제 금지.
- **누락 필드는 마이그레이션이 아니라 `validate`에서 기본값으로 보충.** 새 기능 추가로 필드가 늘면, 구버전엔 그 키가 없다 → 기본값 주입. 이것만으로 "마이너 추가"는 버전 올릴 필요 없음. 버전을 올리는 건 **구조 변경·의미 변경**(예: 통화 단위 변경, 키 이름 변경, Decimal 표현 방식 변경) 때만.
- **Decimal 표현 변경은 반드시 새 버전 + 마이그레이션.** break_eternity의 직렬화 문자열 포맷이 바뀌거나 break_infinity→break_eternity 교체 시 구 세이브의 수치 문자열을 새 `Decimal`로 재파싱하는 마이그레이션 필수(research §1: 드롭인 호환이지만 직렬화 표현은 명시 관리).
- **마이그레이션 실패는 데이터를 버리지 않는다.** 변환 중 예외 → 원본 raw 세이브를 `*.corrupt.bak`로 보존하고 사용자에게 알림. 절대 침묵 삭제 금지.

### 1.4 인코딩·직렬화 규칙(고정)

- **lz-string 인코딩 방식은 하나로 통일**(혼용 시 복호화 실패 — research gotcha). 매체별로:
  - localStorage 저장: `compressToUTF16`(localStorage에 가장 효율적).
  - 파일(`fs`)/내보내기 문자열: `compressToBase64`(URL·텍스트 안전).
  - **한 매체 안에서는 절대 섞지 않는다.** 매체 추상화(§1.6)가 이 규칙을 강제.
- **break_eternity `Decimal`은 JSON 직렬화 안 됨**(research §1 gotcha). 저장 시 `.toString()`, 로드 시 `new Decimal(str)`. 전 스키마에 흩어진 Decimal을 일일이 처리하지 않도록 **직렬화 경계에서 일괄 변환**(toJSON 커스터마이징 또는 reviver/replacer). 이 변환 규칙은 세이브 모듈 한 곳에 캡슐화.
- **특수값 방어**: `Infinity`/`NaN`/`Set`은 JSON 표준이 아니므로 명시 처리(AD serializer 선례: Infinity→문자열, Set→Array). 게임수치엔 native Infinity가 나오면 안 되지만(§2), 방어선으로 둔다.

### 1.5 저장 매체 선택 + 근거

| 매체 | 채택 | 근거 / 트레이드오프 |
|---|---|---|
| **localStorage + lz-string** | 웹 빌드 기본 | 가장 단순·동기·즉시. ~5MB 한계지만 우리 세이브는 압축 후 수십~수백 KB(스파스 ID 집합 + 소수 Decimal)라 여유. 프로토타입~itch.io 배포 단계 |
| **IndexedDB** | 웹 빌드 fallback | localStorage 용량 초과·차단(시크릿 모드 등) 시. 비동기·수 GB. 우리 규모엔 거의 불필요하나 안전망 |
| **`fs` 파일 (NW.js)** | **Steam 빌드 기본** | Steam Cloud 동기화 경로에 직접 기록 → 크로스 PC 싱크 자동. **웹 localStorage와 NW.js localStorage는 별개 저장소**라 fs를 쓰지 않으면 웹↔데스크톱 세이브 이전 불가(research gotcha). **그래서 데스크톱은 처음부터 fs** |
| Steam Cloud | Steam 빌드 동반 | fs 경로를 Cloud에 매핑(계정당 ~100MB, 자동). `lastSave` 교차검증 소스(§1.7)로도 활용 |
| 암호화 저장 | **기각** | 접근성 우선 장르, 변조는 체크섬으로 탐지로 충분. 암호화는 키 관리·디버깅 비용만 추가하고 클라이언트 사이드에선 어차피 우회 가능 |

**결정: 매체를 직접 호출하지 않고 추상화한다(§1.6).** 동일 게임 코드가 웹(localStorage)과 Steam(fs)에서 그대로 돌게 하는 것이 웹→Steam 이식을 막지 않는 핵심.

### 1.6 저장소 추상화 레이어(StorageAdapter 개념)

게임 로직은 "세이브해줘 / 불러와줘"만 호출하고, **어디에 어떻게 저장하는지는 어댑터가 결정**한다. 어댑터 3종(LocalStorage / IndexedDB / FileSystem)이 동일 인터페이스(`read`, `write`, `exists`, `backup`)를 구현한다.

- 빌드 타깃(웹/Steam)에 따라 부팅 시 적절한 어댑터를 주입.
- §1.4 인코딩 규칙(UTF16 vs Base64)을 어댑터가 책임 → 게임 코드는 인코딩을 모름.
- **이식성 효과**: NW.js 패키징 시 FileSystem 어댑터로 교체만 하면 됨. 게임 로직 0줄 수정.

### 1.7 변조 탐지(시계조작·세이브 편집)

economy §3.3이 정한 방어를 **두 층**으로 구현한다(economy는 수식만, 본 문서가 구조 확정):

1. **세이브 무결성(편집 탐지)**: 봉투 `checksum`이 `data`와 불일치하면 → 손상/편집으로 간주, 자동 백업본(§1.8)으로 폴백 시도.
2. **시간 무결성(시계조작 탐지)**: `lastSave` timestamp를 **세이브 파일과 Steam Cloud 양쪽에 기록**하고, 오프라인 계산 시 **둘 중 최댓값**을 `last_save`로 채택(가장 보수적 = elapsed 최소). 클라이언트 `Date.now()`를 미래로 돌려도 이전에 기록된 `lastSave`가 더 크면 음수 elapsed → 0 클램프. 추가로 economy §3.1의 **48h 하드 탬퍼 클램프**가 최종 안전망(시계를 +1년 돌려도 최대 48h만 인정).
   - 순수 오프라인 게임이라 시계조작 치명도는 낮지만(research §3 "접근성 우선"), 48h 클램프 + lastSave 최댓값 채택의 **2중 방어**로 무한 자원 익스플로잇만 확실히 차단.

### 1.8 자동 백업·다중 슬롯

- **롤링 백업**: 매 N회 저장마다 직전 세이브를 `slot.bak.1 → .2 → .3` 식으로 회전 보존(최소 2~3세대). 메인 세이브 손상 시 자동 복구 후보.
- **저장 원자성**: 파일 저장은 "임시 파일에 쓰고 → 성공 시 원자적 교체" 패턴(쓰는 도중 크래시로 반쪽 파일이 메인을 덮어쓰는 것 방지).
- **수동 내보내기/가져오기**: Base64 문자열로 세이브를 클립보드 복사/붙여넣기(유저 백업·기기 이전·버그 리포트용). 봉투 포맷 그대로 사용.

### 2.1 라이브러리 결정: **break_eternity.js** (확정)

| 후보 | 판정 | 근거 |
|---|---|---|
| **break_eternity.js** | **채택** | 테트레이션 10^^1e308까지 + 극소 10^-(10^^1e308). 우리는 **두 방향 모두** 필요: `C/lifetime_C`는 위로 폭발(재하강 누적), `r`은 아래로 0을 향함. break_infinity와 드롭인 호환이라 위험 없음 |
| break_infinity.js | 프로토타입 임시 후보 | 1e(9e15) 한계. dec26 단일런(C≈1.2e40)은 버티지만 **재하강 누적 lifetime_C가 한계 돌파 가능**(economy §5.3 명시) → 결국 교체 필요. API 동일하니 초기엔 써도 되지만, **처음부터 break_eternity 권장**(교체 마이그레이션 비용 회피) |
| decimal.js | 기각 | 임의정밀도라 과한 정밀도 + 느림. 인크리멘탈엔 magnitude만 필요 |
| OmegaNum.js | 기각 | 펜테이션+는 우리 스케일에 과잉(개발자 본인 "Incremental Unlimited급 아니면 쓰지 말라") |

> 설치 주의: 패키지명에 점 포함(`break_eternity.js`). TS 타입 제공.

### 2.2 적용 범위 — 어디서 native 금지

**철칙: 게임의 "수치적 진실"은 전부 `Decimal`. native `number`는 오버플로 위험이 0인 곳에서만.**

| 분류 | 타입 | 예 |
|---|---|---|
| **Decimal 필수(native 절대 금지)** | `Decimal` | `E`, `C`, `D_current`, `D_lifetime`, `lifetime_C`, `QF`, 8단 체인 생산량/비용, `production_mult`·`holographic_mult` 등 모든 배율의 곱 결과, `dec`/`r` 계산의 중간·최종값 |
| **native number 허용** | `number` | 압축기 **보유 개수**(`bought`, 정수 카운트 — 수백~수천 규모), 도전과제/통계 **정확 카운터**(research gotcha: magnitude 우선 라이브러리는 정확 정수 카운팅에 부적합 → 일반 정수 병행), `dt`/타이머/프레임, UI 좌표, 설정값, 배열 인덱스 |
| **경계 주의** | 혼용 지점 명시 | `bought`(number) → 비용 `cost_k(n)=base_k·growth_k^n`(Decimal) 계산 입력. 지수 `n`은 number, 결과는 Decimal. 이 경계를 함수 시그니처에서 명확히 |

**왜 이 경계인가**: `bought`는 native float가 절대 못 넘는 범위(정수 수천)지만, 그 `bought`로 계산하는 **비용·생산량은 즉시 폭발**하므로 Decimal. "개수는 작아도 그 개수가 만드는 값은 천문학적"이 인크리멘탈의 본질. 반대로 도전과제 "압축 1,000,000회" 같은 **정확 카운팅**은 Decimal의 mantissa(약 17자리) 정밀도로는 큰 값에서 1씩 증가가 뭉개질 수 있어(research gotcha) native 정수가 더 정확.

**금지 패턴(코드리뷰 게이트)**: 게임수치에 `+`, `*`, `Math.pow`, `parseFloat`, `Number()` 직접 사용 금지. 반드시 Decimal 메서드(`.add`, `.mul`, `.pow`, …). 이건 game-programmer가 린트 규칙/리뷰로 강제할 항목.

### 2.3 표시 변환(과학·접두사 표기) — 책임 경계

**계산과 표시는 완전히 분리한다.** Decimal은 내부 계산 전용, 화면 출력은 **포맷 레이어**가 전담.

- **포맷 레이어(format 모듈)의 책임**: `Decimal → 사람이 읽는 문자열`. 과학표기(1.23e45), 공학표기, 접두사표기(K/M/B/T/… 또는 게임 테마 접두사), 그리고 우리 게임 특유의 **극소 표기**(`r`은 1e-35 같은 아주 작은 값 → "플랑크 길이의 N배" 같은 테마 변환). 표기법은 유저 설정(§1.1 `settings`).
- **포맷 레이어가 아닌 곳의 책임 아님**: 게임 로직·tick은 절대 문자열 포맷을 다루지 않는다. UI 컴포넌트(Svelte)는 Decimal을 받아 format 모듈을 통해 표시만.
- **"작아짐=강해짐" 화면 역설의 기술적 책임 분담**(GDD 필러①·§15): 내부 `C`(Decimal, 커짐) ↔ 화면 `r`(Decimal, 작아짐)의 변환은 **economy가 정한 매핑 `r = r₀·10^(-dec)`**(파생 계산), 그 결과를 format이 "작은 수 + 큰 글로우"로 표시. **수식은 economy, 계산 구조는 본 문서(파생·비저장), 시각 연출은 graphics-programmer, 문자열 포맷은 format 모듈.** 경계 명확.
- **AD 선례**: `format.js`/`math.js`를 core에서 분리(open-source-games §2). 동일 구조 채택.

### 2.4 직렬화 (세이브와의 접점)

§1.4와 동일 — `Decimal`은 저장 시 `.toString()`, 로드 시 `new Decimal()`. 이 변환은 **세이브 모듈의 직렬화 경계에서 일괄 처리**하고, 게임 로직은 항상 살아있는 Decimal 인스턴스만 본다. break_eternity의 toString 포맷이 향후 바뀌면 §1.3 마이그레이션으로 흡수.

## 3. 오프라인 계산

economy §3이 수식을 **이미 확정**했다(CAP=24h, modifier=0.65, 상전이 직후 1.0, LB=0.5, 48h 탬퍼 클램프). 본 문서는 **그 수식을 어떤 구조로 실행하는가**만 정한다. 수식 재설계 절대 금지.

### 3.1 계산 접근: 일괄(기본) + 미니 시뮬(정밀 모드)

economy §3.1이 명시: **"등가-시간 일괄지급은 보수적 하한"**이며 frozen rate로 modifier·Δt만큼 진행시키는 것과 등가. 두 모드를 제공하되 기본은 일괄.

| 모드 | 방식 | 채택 위치 |
|---|---|---|
| **일괄 지급(기본)** | 접속 끊긴 시점의 생산 rate × 유효시간을 `E`·`C`에 한 번에 더함. economy §3.1 그대로 | **기본값.** 생산이 C에 선형이므로 정확한 하한. 즉시 계산(블로킹 0에 가까움) |
| **미니 시뮬(정밀 모드)** | 유효시간을 500~1000 서브틱으로 분할 재생 → 복리·구매 타이밍 일부 반영 | 설정(§1.1)에서 on. **Web Worker로 오프로드**(메인 스레드 블로킹 방지). research §3 권장 |

**왜 둘 다**: 일괄은 빠르고 안전하지만 오프라인 중 자동구매/복리를 무시(약간 손해). 미니 시뮬은 더 정확하지만 CPU 비용. economy가 "일괄=하한"이라 했으므로 **기본 일괄로 충분**하고, 정밀 모드는 옵션. **단, 미니 시뮬도 틱 수 상한(≤1000)**으로 CPU 스파이크 차단(research §3 gotcha).

### 3.2 탬퍼 클램프(48h) 적용 지점

economy §3.1 공식의 클램프를 **계산 진입 직전 단일 지점**에서 적용한다(여러 곳에 흩어지면 우회 위험):

```
적용 순서(economy §3.1 그대로, 본 문서는 "어디서"만):
  raw_elapsed   = now − last_save              // last_save = §1.7 최댓값 채택
  elapsed_clamp = min(raw_elapsed, CAP·TAMPER_MULT)   // = min(_, 48h)  ← 탬퍼 클램프 지점
  capped        = min(elapsed_clamp, CAP)             // = min(_, 24h)
  long_idle     = 1 + LB·log10(1 + max(0, elapsed_clamp − CAP)/CAP)
  effective_s   = capped · modifier · long_idle       // modifier=0.65 (상전이 직후=1.0)
```

- **클램프는 오프라인 계산 함수의 입구에서 딱 한 번.** 그 함수만이 `last_save`와 `now`를 비교한다 → 시계조작 방어선이 하나로 수렴.
- `last_save`는 §1.7대로 **파일·Cloud 기록의 최댓값**(음수 elapsed → 0).

### 3.3 상전이 직후 modifier=1.0 처리

economy §3.1·§3.2: **상전이(빅 크런치 포함) 직후 첫 오프라인 런은 modifier=1.0**(복귀 임팩트 +54%).

- **상태 플래그로 구현**: `prestige` 네임스페이스(§1.1)에 "마지막 상전이 후 아직 오프라인 보너스 미소비" 플래그를 둔다. 오프라인 계산 시 이 플래그가 켜져 있으면 modifier=1.0 적용 후 **플래그 해제**(1회성).
- **상전이 직후 modifier=1.0과 "상전이 직후 첫 런 mult=1.0"은 별개**(economy §1.2·§7.2). 전자는 *오프라인 효율*, 후자는 *production 부스트 기준점*. 혼동 금지 — 둘 다 "상전이 직후"라는 말을 쓰지만 다른 값. 본 플래그는 **오프라인 modifier 전용**.

### 3.4 능동 메커니즘의 오프라인 처리

systems §6 미해결 질문 #10: 위상 겹침·스핀 네트워크·불확정 요동 등 능동 메커니즘이 오프라인 중 어떻게 동작하나. **아키텍처 차원의 규칙**(구체 수치는 economy/systems):

- **오프라인 중 능동 메커니즘은 "평균/기본값"으로 동작**(systems §2-A "방치 시 기본 공명 자동", §2-E "랜덤 분포로 수렴"의 일반화). 즉 오프라인 rate 계산에 들어가는 메커니즘 배율은 **방치 기본 배율**(최댓값 아님 — 익스플로잇 방지, 최악값 아님 — 불쾌 방지).
- 메커니즘 모듈은 `getIdleBaselineMultiplier()` 같은 **"방치 기본 기여" 질의 인터페이스**를 노출하고, 오프라인 계산기는 그 값으로 rate를 구성. **메커니즘별 오프라인 공식을 오프라인 계산기가 알 필요 없음**(역전된 의존: 메커니즘이 자기 방치값을 안다).
- 불확정 요동(§2-H, 하향 요동 존재)처럼 음의 이벤트가 있는 메커니즘도 **오프라인 중엔 장기평균(양수, systems §2-H "장기 평균 양")**으로 처리 → 자리 비웠는데 손해 보는 불쾌감 차단.

## 4. 게임 루프 아키텍처

방치형은 **장시간(수 주) 끊김 없이 도는 것**과 **표현이 로직을 절대 막지 않는 것**이 생명. AD `intervals.js`(독립 인터벌 분리) + Evolve(Web Worker 루프 분리)의 선례를 따른다.

### 4.1 tick(로직) / render(표현) 분리 원칙

**두 루프는 독립이며 서로의 주기를 모른다.**

| 루프 | 구동 | 주기 | 책임 |
|---|---|---|---|
| **tick (로직)** | 고정 timestep 누산기(accumulator) | 고정 dt(예: 초당 N틱, economy 시뮬이 dt=1~2s 가정이므로 그와 정합되는 결정적 스텝) | 생산·비용·상전이 판정·메커니즘 상태 갱신·자원 변화. **결정적**(같은 입력→같은 결과, 세이브/오프라인 재현성) |
| **render (표현)** | `requestAnimationFrame` | 디스플레이 주사율(가변, 60/120/…) | Svelte 반응성으로 상태→화면. 파티클/VFX(graphics-programmer). **로직 안 함** |

- **고정 timestep 핵심**: render가 느려도(저사양·백그라운드) 로직은 정확한 게임시간으로 진행. 프레임드랍이 진행속도를 바꾸지 않음. economy의 결정적 시뮬과 일치 → 페이싱 검증값이 실제 게임에서 재현됨.
- **누산기 패턴**: 실제 경과시간을 모아 고정 dt 단위로 tick을 0~N회 실행(catch-up). **catch-up 상한**을 둬서 긴 멈춤 후 폭주 방지(Evolve `fastLoop` catch-up 상한 선례). 상한 초과분은 §3 오프라인 경로로 넘김.
- **render는 tick 결과를 읽기만**. 절대 상태를 변경하지 않음(단방향 데이터 흐름).

### 4.2 독립 스케줄러 분리

AD `intervals.js`처럼 **서로 다른 주기의 작업을 독립 객체로 분리**(start/stop/restart 가능):

| 스케줄러 | 주기(개념) | 작업 |
|---|---|---|
| gameLoop | 매 프레임(rAF) → 내부에서 고정 dt tick 누산 | §4.1 로직 |
| autoSave | 수십 초~분 | 세이브 직렬화+기록(§1). **tick과 분리**해 저장 비용이 로직을 끊지 않음 |
| cloudSync | 분 단위 | Steam Cloud 확인/기록(Steam 빌드만) |
| everySecond | 1초 | 저빈도 UI 갱신·통계·`lastSave` 갱신 |

- 저장을 별도 스케줄러로 빼는 이유: 큰 세이브 직렬화가 tick 한 프레임에 끼면 hitch. 빈도·타이밍을 독립 제어.

### 4.3 상태 관리 방식 — 중앙 단일 상태 + 이벤트 버스

- **중앙 단일 상태 컨테이너**(AD `player.js` / Evolve `global` 선례). §1.1 스키마가 곧 이 상태의 형태. **단일 진실 소스** — 모든 모듈이 같은 상태를 읽고, 변경은 명확한 경로로만.
- **이벤트 버스(event-hub)**: "상전이 발생", "입자 발견", "층 진입" 같은 도메인 이벤트를 발행/구독(AD `event-hub.js` 선례). 도감·내러티브·사운드·VFX가 게임로직에 직접 결합하지 않고 이벤트로 반응 → **모듈 경계 분리**(narrative 트리거/codex 해금/사운드가 tick 코드를 오염시키지 않음).
- **파생 캐시**: 비용/생산량 등 매 틱 재계산이 비싼 파생값은 **캐시 + 무효화**(AD `cache.js` 선례). 입력(보유 개수·QF 등)이 바뀔 때만 재계산. 단 **캐시는 저장 안 함**(§1.1) — 항상 상태에서 재생성 가능해야 함.

### 4.4 모듈/폴더 구조 개요 (폴더 레이아웃 수준)

AD `src/core/` 기능별 분리(open-source-games §2)를 우리 도메인에 매핑. **이것은 청사진 수준이며, 정확한 파일명/분해는 game-programmer 재량.**

```
src/
  core/
    state/            // 중앙 상태 컨테이너 + 스키마 타입(§1.1)
    loop/             // 고정 timestep tick 누산기 + 스케줄러들(§4.1·4.2)
    bignum/           // break_eternity 래퍼·상수·가드(§2). Decimal 진입점 단일화
    format/           // Decimal→표시 문자열, 과학/접두사/극소 표기(§2.3)
    save/             // 봉투·직렬화·압축·마이그레이션·체크섬·StorageAdapter(§1)
      adapters/       //   localStorage / indexeddb / filesystem(§1.6)
    offline/          // 오프라인 계산기 + 미니시뮬 워커(§3)
    chain/            // 8단 압축기 체인 생산/비용(economy §2 수식 소비)
    prestige/         // 상전이·QF·빅 크런치·재하강 회차/집중층(economy §1, systems §2-K·§5)
    layers/           // 층 진입 판정·서브층 벽(systems §1)
      mechanics/      //   층별 메커니즘 모듈, 각자 serialize/idleBaseline 인터페이스(§1.1·§3.4)
    codex/            // 입자 도감: 발견 판정·보너스(데이터 주도, AD secret-formula식)(systems, codex §13)
    research/         // 연구 트리: 노드 구매·효과 적용
    events/           // 이벤트 버스(§4.3)
  data/               // 게임 데이터(입자 정의 87·연구 노드 52·층 정의·비용 파라미터)
                      //   ← 로직과 분리(데이터 주도). economy/content 산출 수치가 여기 채워짐
  ui/                 // Svelte 컴포넌트(탭·체인·도감·상전이·설정). 상태 읽기 전용
  render/             // 파티클/VFX/스케일 줌(graphics-programmer 영역)
  platform/           // 빌드 타깃 분기: 웹 vs NW.js. steamworks.js 연동(§5)
```

- **`data/` 분리가 핵심**(AD `secret-formula/` 선례): 입자·연구·층·비용을 **코드가 아닌 데이터**로 두면, economy/content/systems가 확정한 수치·정의를 game-programmer가 로직 수정 없이 주입·튜닝. 밸런싱 이터레이션(Vite HMR)과 직결.
- **`mechanics/`의 각 모듈은 자기완결**: 직렬화(§1.1)·방치기본값(§3.4)·이벤트 발행을 스스로 책임 → 세이브/오프라인/루프 코어가 메커니즘 내부를 모름. "한 층=한 새로움"(필러④)을 **모듈 추가만으로** 확장.

## 5. 웹→Steam 패키징

### 5.1 결정: **NW.js + steamworks.js** (확정)

GDD §11·research §5에서 확정. 본 문서는 **근거 + 대안 기각 사유 + 빌드 경로**를 정리.

| 후보 | 번들/메모리 | Steam 오버레이 | Steam SDK | 판정·근거 |
|---|---|---|---|---|
| **NW.js** | ~80–150MB / ~150MB | **작동** | steamworks.js(npm, 컴파일 불필요) | **채택.** Steam 게임 5700+ 사용 실적. **단일 JS 컨텍스트**(프론트/백 분리 불필요)가 인크리멘탈의 단일 상태·단일 루프 구조에 자연스러움. Node `fs` 직접 접근 → Steam Cloud 경로 세이브(§1.5) |
| Electron | ~100–150MB / ~150–300MB | 작동 | steamworks.js 가능 | **기각(차점).** 오버레이·SDK는 동등하나 **IPC 브리지로 프론트/백 분리 필수** → 단일 컨텍스트인 우리 구조에 불필요한 복잡도. 생태계는 최대지만 우리에겐 NW.js 이점이 큼 |
| Tauri | ~5–10MB / ~30–50MB | **미지원(#6196 "not planned")** | 바인딩 없음(Rust 크레이트 직접 작성 필요) | **기각(치명).** 번들이 가장 작지만 **WebView2가 그래픽 디바이스 후킹을 차단 → Steam 오버레이 영구 불가**. Steam 게임에 오버레이 부재는 치명적. steamworks.js도 미지원 → Steam 연동에 막대한 Rust 공수. **번들 크기에 현혹되지 말 것**(research gotcha) |

**번들 크기 트레이드오프 수용**: NW.js의 ~150MB는 Tauri의 ~10MB보다 크지만, **(1) Steam 오버레이라는 필수 기능을 얻고, (2) 우리 게임 코드(Svelte ~1.6kb + 로직)는 패키지에 거의 기여 안 하므로** 추가 부담은 사실상 Chromium 런타임 한 벌. 방치형 PC 게임에서 150MB는 무시 가능한 크기. **오버레이>크기.**

### 5.2 Steamworks 연동 경계

- **steamworks.js**(research §5-2): `npm i` 만으로 사용, 네이티브 컴파일 불필요. 업적·플레이어 정보·오버레이.
- **연동 코드는 `platform/`에 격리**(§4.4): 게임 로직은 Steam API를 직접 부르지 않고 **추상 인터페이스**(예: "업적 해금 요청")로만 호출. 웹 빌드에선 이 인터페이스가 no-op. → **웹 우선 개발이 Steam 의존으로 오염되지 않고, Steam 이식이 게임 로직을 안 건드림**.
- **업적 ↔ 도감/상전이 매핑**: codex 발견·상전이 이벤트(§4.3 이벤트 버스)를 구독해 Steam 업적으로 변환(plat 레이어). 게임 코어는 Steam을 모름.
- 대안 기각: greenworks(C++ 컴파일 의존, 유지보수 수동적), steamworks-ffi-node(가능하나 steamworks.js가 더 표준적·간단).

### 5.3 빌드 파이프라인 개요

```
개발:    Vite dev server + HMR (Svelte/TS, 밸런싱 이터레이션)
                │
빌드:    Vite build → 정적 번들(HTML/JS/CSS)  ← 웹·데스크톱 공통 산출
                │
        ┌───────┴────────┐
   웹 배포            데스크톱 패키징
   (itch.io,         정적 번들 + NW.js 런타임 + package.nw
   galaxy.click)     + steamworks.js → Win/(Mac/Linux) 실행파일
                          │
                     Steam 업로드(Steamworks SDK / steamcmd)
```

- **공통 정적 번들**: Vite 빌드 산출물 하나가 웹과 데스크톱 양쪽에 들어감 → 코드 분기는 `platform/`의 어댑터 주입(StorageAdapter §1.6, Steam 인터페이스 §5.2)으로만. **이식성 보장의 구조적 근거.**
- **소스 노출 인지**(research gotcha): NW.js 기본 패키징은 JS 소스가 `package.nw`(ZIP)에 노출. 치팅 방지가 필요하면 `nwjc` 바이트코드화 옵션 있으나, **접근성 우선 장르라 기본 생략**(필요 시 후속 결정).
- 중간 배포(itch.io 등)는 웹 번들 그대로 → Steam 이전에 플레이테스트 채널 확보(GDD §2).

## 6. 성능 예산

방치형 특유의 4대 부하: **(A) 장시간 무중단 구동, (B) dec40+ 큰 수 연산, (C) 다수 입자 렌더, (D) 탭 백그라운드/복귀.**

### 6.1 장시간 구동(메모리·누수)

- **목표**: 수 일~수 주 탭/앱을 켜둬도 메모리 안정(누수 0), 진행 정확.
- **누수 방지 원칙**: 이벤트 버스 구독 해제 규율(특히 상전이·층전환으로 메커니즘 모듈이 교체될 때 구버스 구독 정리), 파티클 객체 풀링(§6.3), 무한 증가 배열 금지(통계·로그는 상한/회전).
- **결정적 tick의 부수효과**: 고정 timestep이라 장시간에도 누적 부동소수 드리프트가 게임시간을 왜곡하지 않음(누산기 기반).
- **자동 저장이 장기 구동의 안전벨트**: 크래시/메모리 한계로 탭이 죽어도 autoSave(§4.2)가 최근 상태 보존.

### 6.2 dec40+ 큰 수(연산 비용)

- **break_eternity는 native보다 느림**(곱셈류 수배~). 따라서 **매 틱 Decimal 연산 횟수를 최소화**:
  - 파생값 캐시+무효화(§4.3)로 변하지 않는 비용/배율을 재계산 안 함.
  - 8단 체인 tick은 AD식 역순 단일 패스(open-source-games §2)로 티어당 상수 회 연산.
  - 닫힌형 등비급수로 "한 번에 N개 구매" 비용 계산(economy도 `while-bank` 루프 금지 — economy §0 주석) → 루프 대신 공식.
- **표시 갱신은 저빈도**: 화면 숫자는 매 프레임이 아니라 everySecond(§4.2)급으로 충분(인간이 읽는 속도). format(§2.3) 비용을 rAF에서 분리.

### 6.3 다수 입자 렌더(graphics-programmer 연계 지점)

- **책임 경계**: 본 문서는 **로직↔렌더 분리 계약**만 정한다. 실제 파티클 시스템·셰이더·스케일 줌은 **graphics-programmer**(Canvas2D/WebGL, 60fps).
- **연계 인터페이스**: render는 tick 상태를 **읽기 전용**으로 받음(§4.1). 파티클 수는 **표현 파라미터**(graphics)이지 게임 상태가 아님 → 입자 수를 줄여도 로직/진행 불변.
- **아키텍처가 보장할 것**: (1) render가 막혀도 tick 정확(§4.1 고정 timestep), (2) 파티클은 객체 풀링 전제(누수·GC 스파이크 방지 — §6.1), (3) 저사양/백그라운드에서 파티클 LOD/스킵을 render가 자율 결정(로직 영향 0). 구체 수치(파티클 상한·LOD 단계)는 graphics-programmer.

### 6.4 탭 백그라운드/복귀

- **문제**: 브라우저는 백그라운드 탭의 rAF·타이머를 스로틀/정지 → render·tick 누산이 멈추거나 느려짐.
- **결정: 백그라운드는 오프라인 경로로 수렴.** 탭이 백그라운드/숨김이 되면(visibility 변화) 정밀 실시간 tick을 포기하고, **복귀 시 경과시간을 §3 오프라인 계산으로 일괄 반영**. 즉 "백그라운드 = 짧은 오프라인". 이로써:
  - 백그라운드에서 CPU 안 씀(배터리·발열),
  - 복귀 시 §3 공식(48h 클램프·modifier)으로 일관 처리 → **백그라운드를 통한 익스플로잇도 동일 클램프로 차단**,
  - 실시간 tick의 catch-up 폭주(§4.1 상한 초과분)도 같은 경로로 흡수.
- **NW.js(데스크톱)**: 백그라운드 스로틀이 브라우저보다 덜하지만, **동일 정책**(visibility/포커스 기반 오프라인 전환) 적용해 웹과 거동 일치 → 플랫폼별 분기 최소화.
- **경계 케이스**: 짧은 탭 전환(수초)까지 오프라인 처리하면 부자연스러우므로, **임계시간 이상 백그라운드일 때만** 오프라인 전환(임계값은 튜닝 파라미터). 그 이하는 복귀 시 짧은 catch-up tick.
